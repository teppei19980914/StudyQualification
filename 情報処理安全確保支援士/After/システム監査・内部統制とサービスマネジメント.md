## 監査とは何か

### 監査の定義

- **システム監査基準(経済産業省)**
- **内部監査基準(IIA：内部監査人協会)**

監査とは、**一定の基準に照らして対象を独立した立場で評価し、その結果を利害関係者に伝える活動**です。

### 監査の本質

監査の本質は以下の3点に集約されます。

1. **独立性**
2. **客観性**
3. **証拠に基づく評価**

→ 「問題を見つけること」自体が目的ではなく、**組織の健全性/信頼性を高めること**が最終目的です。

## 助言型監査(Advisory Audit)

### 定義(エビデンス：内部監査基準/システム監査基準)

助言型監査とは、**監査対象の現状を評価した上で、改善に向けた助言/提案を行う監査**です。
つまり、「こうすべきである」と**改善方向を示す**監査です。

### 助言型監査の特徴

| 観点 | 内容 |
|---|---|
| 主目的 | 改善/高度化の支援 |
| 監査結果 | 助言/提言が中心 |
| 法的拘束力 | なし |
| 利用場面 | 成熟度向上、改善フェーズ |

### 助言型監査の具体例

- セキュリティ対策が不十分な部門に対し
  → **「多要素認証導入が望ましい」**
- クラウド利用に関して
  → **「責任分界点を明確にすべき」**

→ **「改善のヒントを与える」**のが助言型監査です。

### 試験でのキーワード(助言型)

- 改善提案
- 助言
- 支援
- コンサルティング的要素

※ ただし **コンサルそのものではない**点に注意が必要です。

## 保証型監査(Assurance Audit)

### 定義(エビデンス：内部監査基準)

保証型監査とは、**監査対象が一定の基準を満たしているかどうかについて、保証(お墨付き)を与える監査**です。
つまり、「基準に適合している／していない」を明確に示す監査です。

### 保証型監査の特徴

| 観点 | 内容 |
|---|---|
| 主目的 | 適合性の保証 |
| 監査結果 | 意見/評価(適合／不適合) |
| 利害関係者 | 経営層/外部ステークホルダ |
| 利用場面 | 信頼性確保、説明責任 |

### 保証型監査の具体例

- ISMS監査
  → **ISO/IEC 27001に適合しているか**
- 内部統制監査
  → **J-SOXに準拠しているか**

→ **「第三者に説明するための監査」**が保証型です。

## 助言型監査と保証型監査の比較

| 観点 | 助言型監査 | 保証型監査 |
|---|---|---|
| 目的 | 改善支援 | 適合性保証 |
| 結果 | 提言/助言 | 評価/意見 |
| 強制力 | なし | なし(ただし影響大) |
| 試験での位置づけ | 改善フェーズ | 評価フェーズ |

**「改善したい」→ 助言型**
**「守れているか示したい」→ 保証型**

## 監査とコンサルの違い

| 観点 | 監査 | コンサル |
|---|---|---|
| 独立性 | 必須 | 必須ではない |
| 役割 | 評価/助言 | 解決策実行支援 |
| 責任 | 判断は被監査側 | 成果責任を負うことも |

## システム監査(エビデンス：経産省「システム監査基準」)

システム監査とは、**情報システムの信頼性/安全性/効率性/リスク管理の適切性を、独立した立場から評価し助言する活動**です。

### システム監査基準の構成(試験頻出)

経済産業省の「システム監査基準」は以下の3つの基準で構成されます。

| 基準 | 内容 |
|---|---|
| **一般基準** | 監査人の資質・独立性・倫理に関する基準 |
| **実施基準** | 監査の計画・実施方法に関する基準 |
| **報告基準** | 監査報告書の作成・報告に関する基準 |

### システム監査基準とシステム管理基準の関係

| 基準 | 役割 | 利用者 |
|---|---|---|
| **システム監査基準** | 監査を行う側の行動規範 | システム監査人 |
| **システム管理基準** | 情報システムの管理のあるべき姿(判断基準) | 被監査部門(管理者) |

> システム管理基準は **監査の判断尺度(ものさし)** として使用されます。

### 監査の手順(プロセス)

```
1. 監査計画の策定(年間計画/個別計画)
2. 予備調査(対象システムの概要把握)
3. 本調査(証拠の収集・分析)
4. 評価・結論(監査意見の形成)
5. 監査報告書の作成・報告
6. フォローアップ(改善状況の確認)
```

### 監査調書(Working Papers)

監査調書とは、**監査の実施過程と結果を記録した文書**です。

| 項目 | 内容 |
|---|---|
| 目的 | 監査意見の根拠を文書化、品質管理 |
| 内容 | 監査手続、収集した証拠、分析結果、結論 |
| 保管 | 一定期間保管が必要(説明責任) |

> 監査調書は **監査報告書の裏付け** であり、試験では「監査調書に記録すべき事項」が問われます。

### 監査人の独立性(試験最頻出)

#### 精神的独立性(Independence of Mind)

- 監査人が**判断において偏りがない**こと
- 内面的な公正さ・客観性

#### 外観的独立性(Independence in Appearance)

- 第三者から見て**独立していると認められる**こと
- 監査対象の部門に所属していない等の外形的要件

> 試験では「**自部門の監査はできない(外観的独立性の欠如)**」が頻出です。

#### 独立性に関する具体的な制約

| 状況 | 独立性 | 理由 |
|---|---|---|
| 自部門のシステムを監査 | **不可** | 外観的独立性がない |
| 過去に開発に関与したシステムを監査 | **不可** | 精神的独立性に疑義 |
| 監査部門が情報システム部門から独立 | **可** | 組織的独立性が確保 |

## 内部監査との違い

| 種別 | 対象範囲 | 特徴 |
|------|-----------|-------|
| 内部監査 | 組織全体(財務/業務/法務/IT) | 経営者に対する保証 |
| システム監査 | ITに特化 | 技術的観点からの評価 |

内部監査の一部としてシステム監査が実施されることも多いです。

## 財務報告に係る内部統制の評価及び監査に関する実施基準

### 制度の位置づけと背景

財務報告にかかる内部統制の評価及び監査に関する実施基準です。
通称「J-SOX 実施基準」とも呼ばれます。

### 根拠法令

* 金融商品取引法 第24条の4の4
* 企業会計審議会 内部統制部会
* 金融庁公表資料

### 制度導入の背景

* 粉飾決算/不正会計の多発(国内外)
* 米国 SOX法(サーベンス/オクスリー法) を参考に導入
* 投資家保護/市場の信頼性確保が目的

### 制度の目的

J-SOX の目的は「IT統制そのもの」ではなく、あくまで「財務報告の信頼性を確保すること」です。

### 重要ポイント

* 業務効率化が目的ではない
* セキュリティ対策一般でもない
* 財務報告に影響する内部統制のみが対象

### 全般統制と業務処理統制

ITを取り入れた情報システムに関する統制は「全般統制」と「業務処理統制」の2つから構成されます。

#### 全般統制

業務処理統制が有効に機能する環境を保証するための統制活動を意味しており、通常、複数の業務処理統制に関係する方針と手続を言います。

#### 業務処理統制

業務を管理するシステムにおいて、承認された業務が全て正確に処理、記録されることを確保するために業務プロセスに組み込まれたITに係る内部統制です。

## 内部統制(COSO)

### 内部統制の定義

内部統制とは、業務の有効性/効率性/財務報告の信頼性/事業活動に係る法令遵守/資産の保全を確保するために整備/運用される仕組みです。

### 目的

1. 業務の有効性/効率性
2. 財務報告の信頼性
3. 法令遵守
4. 資産の保全

### 構成要素

- 統制環境
- リスク評価
- 統制活動
- 情報と伝達
- モニタリング
- IT統制(拡張要素)

## IT統制(ITGC/Application Controls)

### IT全般統制(ITGC)

- アクセス管理
- 変更管理
- 運用管理
- 開発管理
- 外部委託管理

### 業務処理統制

- 入力チェック
- 計算ロジックの正確性
- 例外処理管理
- 承認/承認ワークフロー

## 改善勧告(システム監査基準)

監査で発見した問題に対し、**改善案を提示して経営層に助言する活動**です。

特徴：

- 法的拘束力はない
- 経営主体の改善行動を促す
- PDCA の「A(Act)」を担う

## 監査証拠

監査を実施する上で最重要要素が **監査証拠(Audit Evidence)** です。

### 監査証拠とは

監査意見の根拠となる、**十分かつ適切な情報**です。

- **十分性(Sufficiency)**：証拠の量
- **適切性(Appropriateness)**：証拠の質(信頼性/関連性)

### 監査証拠の種類

#### 1. 文書証拠(最も信頼性が高い)

- 方針、規程、手順書
- ログ、帳票、証跡
- 契約書、SLA、委託契約

#### 2. 物的証拠

- スクリーンショット
- サーバルーム設備
- バックアップ媒体

#### 3. 証言(インタビュー)

- 担当者への聞き取り
※単独では信頼性が低いため、文書証拠で裏付ける必要があります。

#### 4. 観察

- 作業の実施状況の観察
- 手続の運用状況の立会い確認

## 監査証拠の入手方法(エビデンス：システム監査基準)

1. **インタビュー(聞き取り)**
2. **文書レビュー(規程/手順/ログ等)**
3. **現場観察(作業立会い)**
4. **突合(ログと作業履歴の整合性確認)**
5. **再実行(手続を監査人が再試行)**

これらを組み合わせて **相互補完** することが必須です。

## 監査証拠の信頼性(重要ポイント)

信頼性が高い順序：

1. **第三者が作成した証拠**
2. 組織内部の文書証拠
3. 物的証拠
4. 証言(インタビュー)

試験では必ず以下の原則が問われます。

- **証言だけでは不十分 → 文書証拠で裏付ける必要がある**
- **ログのような"自動生成証拠"は信頼性が高い**
- **複数証拠で相互補完すると証拠の適切性が高まる**

## クラウド導入プロセスに対する監査

クラウド導入は外部委託であるため、監査では以下が特に重要です。

### 1. 導入企画段階

- 導入目的の妥当性
- リスク分析(データ所在、機密分類)
- RTO/RPO要件
- 責任共有モデル(ISO 27017)

### 2. 契約(SLA)監査

- 可用性保証
- インシデント通知
- データ消去要件
- 監査証跡(ログ)の提供
- サブプロセッサ管理

### 3. 構築/移行

- 最小権限設定
- 暗号化適用
- 移行時のデータ保護

### 4. 運用段階

- アカウント管理
- ログ管理
- 委託先管理(SOC2, ISO27001)

## 情報セキュリティ監査

### 情報セキュリティ監査基準(経済産業省)

情報セキュリティ監査基準は、**情報セキュリティに関する監査を行う際の監査人の行動規範**です。

| 基準 | 内容 |
|---|---|
| **一般基準** | 監査人の権限・独立性・能力・注意義務 |
| **実施基準** | 監査計画の策定、監査手続の実施 |
| **報告基準** | 監査報告書の作成、意見表明 |

### 情報セキュリティ管理基準(経済産業省)

情報セキュリティ管理基準は、**組織が情報セキュリティを管理するための実践規範**です。
**ISO/IEC 27001 および 27002** をベースに策定されています。

> 情報セキュリティ監査基準は「監査する側」、情報セキュリティ管理基準は「管理する側(判断尺度)」の基準です。

### 監査基準の体系整理

| 基準名 | 策定者 | 目的 |
|---|---|---|
| **システム監査基準** | 経済産業省 | IT全般の監査の行動規範 |
| **システム管理基準** | 経済産業省 | IT管理の判断尺度 |
| **情報セキュリティ監査基準** | 経済産業省 | 情報セキュリティ監査の行動規範 |
| **情報セキュリティ管理基準** | 経済産業省 | 情報セキュリティ管理の判断尺度 |

## SOC報告書(委託先の内部統制評価)

### SOC報告書とは

SOC(System and Organization Controls)報告書とは、**外部委託先(クラウド事業者等)の内部統制の有効性を、独立した監査人が評価した報告書**です。

### SOC報告書の種類(試験頻出)

| 種類 | 対象 | 内容 | 利用者 |
|---|---|---|---|
| **SOC 1** | 財務報告に係る内部統制 | J-SOX/SOX対応の委託先統制評価 | 委託元の監査人 |
| **SOC 2** | セキュリティ・可用性・処理の完全性・機密性・プライバシー | **情報セキュリティ関連の統制評価** | 委託元組織(限定配布) |
| **SOC 3** | SOC 2 と同じ基準 | SOC 2 の要約版(一般公開可能) | 一般公開 |

### SOC 2 の Type I と Type II

| 種類 | 評価内容 | 特徴 |
|---|---|---|
| **Type I** | ある**時点**における統制の設計の適切性 | スナップショット評価 |
| **Type II** | ある**期間**にわたる統制の運用の有効性 | **Type II のほうが信頼性が高い** |

> 試験では「**SOC 2 Type II = 一定期間にわたる運用の有効性を評価**」が問われます。

### 委託先管理のポイント(午後問題で頻出)

| 管理項目 | 内容 |
|---|---|
| **委託先の選定基準** | セキュリティ体制、認証取得状況(ISO 27001等)の確認 |
| **契約における要件** | セキュリティ要件、SLA、インシデント通知義務、監査権の明記 |
| **再委託の管理** | 再委託先(サブプロセッサ)の把握と承認 |
| **定期的なモニタリング** | SOC報告書の入手、定期監査、セキュリティ評価 |

## ISMS適合性評価制度

### ISMS(Information Security Management System)とは

ISMS とは、**組織の情報セキュリティを体系的に管理するための仕組み**です。

### ISO/IEC 27000 ファミリー(試験頻出)

| 規格 | 内容 |
|---|---|
| **ISO/IEC 27001** | ISMS の**要求事項**(認証基準) |
| **ISO/IEC 27002** | 情報セキュリティ管理策の**実践規範**(ベストプラクティス) |
| **ISO/IEC 27005** | 情報セキュリティ**リスクマネジメント** |
| **ISO/IEC 27014** | 情報セキュリティ**ガバナンス** |
| **ISO/IEC 27017** | **クラウドサービス**の情報セキュリティ管理策 |
| **ISO/IEC 27018** | **クラウド上の個人情報保護** |

### ISMS認証取得プロセス

```
1. 適用範囲の決定
2. 情報セキュリティポリシーの策定
3. リスクアセスメントの実施
4. リスク対応計画の策定(管理策の選定)
5. 適用宣言書(SoA)の作成
6. ISMS の運用
7. 内部監査の実施
8. マネジメントレビュー
9. 認証審査(ステージ1/ステージ2)
10. 認証取得 → 継続的改善(PDCAサイクル)
```

### ISMS の PDCA サイクル

| フェーズ | 内容 |
|---|---|
| **Plan** | ISMS の確立(ポリシー策定、リスクアセスメント) |
| **Do** | ISMS の導入・運用(管理策の実施) |
| **Check** | ISMS の監視・レビュー(内部監査、マネジメントレビュー) |
| **Act** | ISMS の維持・改善(是正処置、予防処置) |

## サービス可用性(Availability)

必要な時に情報及び関連資産にアクセスできる状態を確実にすることです。
つまり、「サービスが止まらない」「利用者が必要な時に利用できる」状態を守ることが大切です。

### 情報セキュリティの三大要素CIAのA

|要素|意味|
|:----|:----|
|Confidentiality（機密性）|情報を秘密に保つ|
|Integrity（完全性）|改ざんされていない状態を保つ|
|Availability（可用性）|必要な時に利用できる|

### 可用性を損なう攻撃

1. DoS/DDoS攻撃
    * 大量リクエストでサーバを高負荷にし、サービスを停止させる
2. リソース枯渇(メモリ/ディスクの消費)
    * ログの肥大化や無限ループ
3. ハード障害
    * HDD障害やNIC断に加え、電源障害など
4. ランサムウェア
    * システム暗号化により利用不可

### 可用性を確保する手段

1. 冗長化(Redundancy)
    1. サーバの二重化
    2. RAID
    3. ネットワークの多重化
    4. DNS冗長
2. 可用性設計(Availability Architecture)
    1. フォールトトレランス
    2. フェイルオーバー
    3. ロードバランシング
    4. クラスタリング
3. バックアップと迅速なリストア
    1. RTO(復旧時間目標)
    2. RPO(復旧時点目標)
    3. バックアップ多重保管
4. 管理とインシデント対応
    1. 障害監視
    2. アラート
    3. SLA違反検知
5. DDoS対策
    1. CDN/WAF
    2. 通信のレート制限
    3. トラフィックフィルタリング
    4. ISPレベルでの対策

## 可用性に関連する周辺知識

1. SLA(Service Level Agreement)
    * 稼働率(例：99.99%)
    * 平均故障間隔(MTBF)
    * 平均修復時間(MTTR)
2. SPOF(Single Point of Failure)
    * システムの単一故障点
    * 可用性設計における最大の敵
3. BCP(事業継続計画)
    * 大規模障害でも業務を継続する計画
    * DR(Disaster Recovery)との関連
4. クラウドの可用性
    * リージョン/ゾーンの冗長化
    * Multi-AZ配置
    * CDNによる配信負荷分散
5. データベース可用性
    * レプリケーション
    * フェイルオーバー
    * シャーディング

### 可用性の指標(試験頻出)

| 指標 | 定義 | 計算式 |
|---|---|---|
| **MTBF(Mean Time Between Failures)** | 平均故障間隔(故障と故障の間の稼働時間の平均) | 総稼働時間 / 故障回数 |
| **MTTR(Mean Time To Repair)** | 平均修復時間(故障から復旧までの時間の平均) | 総修復時間 / 故障回数 |
| **稼働率(Availability)** | システムが正常に利用可能な時間の割合 | **MTBF / (MTBF + MTTR)** |

#### 稼働率の具体例

| 稼働率 | 年間ダウンタイム |
|---|---|
| 99.9%(スリーナイン) | 約8.8時間 |
| 99.99%(フォーナイン) | 約52.6分 |
| 99.999%(ファイブナイン) | 約5.3分 |

### フォールトトレランスとフェールセーフの違い

| 用語 | 説明 | 例 |
|---|---|---|
| **フォールトトレランス** | 障害が発生しても**サービスを継続**する設計 | サーバ冗長化、RAID |
| **フェールセーフ** | 障害発生時に**安全な状態に移行**する設計 | 信号機の故障→全赤表示 |
| **フェールソフト** | 障害発生時に**機能を縮退して稼働**を継続 | CPUの一部故障→性能低下で継続 |
| **フールプルーフ** | 人間の誤操作を**防止する**設計 | 確認ダイアログ表示 |

## BCP/BCM と DR(事業継続)

### BCP(Business Continuity Plan：事業継続計画)

BCP とは、**自然災害・サイバー攻撃・パンデミック等の緊急事態において、事業の継続または早期復旧を図るための計画**です。

### BCM(Business Continuity Management：事業継続マネジメント)

BCM とは、**BCP の策定・運用・見直しを継続的に行うマネジメントプロセス**です。

> BCP は「計画(文書)」、BCM は「マネジメント(プロセス)」です。

### BCP で定義すべき要素

| 要素 | 内容 |
|---|---|
| **BIA(Business Impact Analysis)** | 事業影響度分析(業務停止時の影響を評価) |
| **RTO(Recovery Time Objective)** | 復旧時間目標(いつまでに復旧するか) |
| **RPO(Recovery Point Objective)** | 復旧時点目標(どの時点のデータまで復旧するか) |
| **RLO(Recovery Level Objective)** | 復旧レベル目標(どの水準まで復旧するか) |
| **優先業務の特定** | 最優先で復旧すべき業務の順位付け |

### DR(Disaster Recovery：災害復旧)

DR とは、**災害発生後にITシステムを復旧するための仕組み・体制**です。

#### DRサイトの種類(試験頻出)

| 種類 | 説明 | 復旧時間 | コスト |
|---|---|---|---|
| **ホットサイト** | 本番同等の環境を常時稼働。即座に切り替え可能 | **最短(数分〜数時間)** | 最高 |
| **ウォームサイト** | 機器は設置済みだが、データ同期は定期的。起動・設定が必要 | 数時間〜数日 | 中 |
| **コールドサイト** | 場所・電源・通信回線のみ確保。機器の搬入・設定が必要 | 数日〜数週間 | 最低 |

> 試験では「**ホットサイト=即時切替・高コスト**」「**コールドサイト=低コスト・復旧遅い**」の対比が問われます。

## ITサービスマネジメント

### ITIL(Information Technology Infrastructure Library)

ITIL とは、**ITサービスマネジメントのベストプラクティスをまとめたフレームワーク**です。

### ITサービスマネジメントの主要プロセス(試験頻出)

#### インシデント管理 vs 問題管理(最頻出)

| 項目 | インシデント管理 | 問題管理 |
|---|---|---|
| 目的 | **サービスの迅速な復旧** | **根本原因の究明と再発防止** |
| 対応 | 応急処置(ワークアラウンド) | 恒久対策 |
| 時間軸 | 短期(今すぐ復旧) | 中長期(根本解決) |
| 例 | サーバ再起動で復旧 | メモリリークの原因を特定し修正 |

> 「**インシデント管理=復旧優先**」「**問題管理=原因究明**」の区別は必ず問われます。

#### その他の主要プロセス

| プロセス | 目的 |
|---|---|
| **変更管理** | システム変更に伴うリスクを管理し、承認プロセスを経て実施 |
| **構成管理(CMDB)** | IT資産・構成要素の情報を一元管理 |
| **リリース管理** | 変更のリリース(本番適用)を計画的に実施 |
| **サービスレベル管理** | SLA の策定・監視・見直し |
| **キャパシティ管理** | 将来の需要に備えたリソース計画 |
| **可用性管理** | サービスの可用性目標の達成 |
| **情報セキュリティ管理** | ITサービスにおける情報セキュリティの確保 |
| **サービスデスク** | ユーザからの問い合わせの単一窓口(SPOC：Single Point of Contact) |

### 変更管理のプロセス(午後問題で重要)

```
1. RFC(Request for Change)の提出
2. 変更の評価・影響分析
3. CAB(Change Advisory Board：変更諮問委員会)での審議
4. 変更の承認/却下
5. 変更の実施
6. PIR(Post Implementation Review：実施後レビュー)
```

> **CAB(変更諮問委員会)** は変更管理の重要な意思決定機関であり、試験で頻出です。

### ISO/IEC 20000

ISO/IEC 20000 は、**ITサービスマネジメントシステム(ITSMS)の国際認証規格**です。
ITIL をベースとした要求事項を定めています。

| 比較項目 | ITIL | ISO/IEC 20000 |
|---|---|---|
| 性質 | ベストプラクティス(推奨) | **認証規格(要求事項)** |
| 認証 | 個人資格 | **組織の認証** |
| 強制力 | なし | 認証取得に必要 |

## 全体整理

### 監査関連の基準・制度の体系

| 基準・制度 | 対象 | 策定者/根拠 |
|---|---|---|
| **システム監査基準** | IT全般の監査 | 経済産業省 |
| **システム管理基準** | IT管理の判断尺度 | 経済産業省 |
| **情報セキュリティ監査基準** | 情報セキュリティの監査 | 経済産業省 |
| **情報セキュリティ管理基準** | 情報セキュリティ管理の判断尺度 | 経済産業省 |

| **ISO/IEC 27001** | ISMSの認証基準 | ISO/IEC |
| **ISO/IEC 20000** | ITSMSの認証基準 | ISO/IEC |
| **SOC報告書** | 委託先の内部統制評価 | AICPA |

### 監査の種類と目的の整理

| 監査の種類 | 目的 | 判断基準 |
|---|---|---|
| **助言型監査** | 改善の支援・助言 | システム管理基準等 |
| **保証型監査** | 基準適合の保証 | ISO 27001、J-SOX等 |
| **内部監査** | 組織全体のガバナンス評価 | 内部監査基準 |
| **システム監査** | IT特化の評価 | システム監査基準 |
| **情報セキュリティ監査** | セキュリティの評価 | 情報セキュリティ監査基準 |

### 内部統制の構成要素とIT統制の関係

| 内部統制の構成要素 | IT統制との関連 |
|---|---|
| 統制環境 | 情報セキュリティポリシー、経営方針 |
| リスク評価 | IT リスクアセスメント |
| 統制活動 | アクセス制御、変更管理、運用管理(**ITGC/業務処理統制**) |

| モニタリング | 監査、レビュー、継続的監視 |
| IT統制 | ITGC(全般統制) + 業務処理統制 |
