## サイバーセキュリティフレームワーク(CSF)

組織がサイバーセキュリティリスクを管理・低減するためのガイドライン・ベストプラクティス・標準の集合です。

### CSFの構造

1. **フレームワークコア**
    - 何をするべきかを示す6つの基本機能(Functions)とその下にぶら下がるカテゴリ(Categories)/サブカテゴリ(SubCategories)です。
    - CSF 2.0(2024年2月公開)でGovern機能が追加され、従来の5機能から6機能に拡張されました。
2. **プロファイル**
    - 理想状態(Target Profile)と現状(Current Profile)を比較し、組織の課題を明確化します。
3. **実施階層**
    - 組織のリスク管理の習熟度を4段階で評価します。

### 基本機能(Functions)

1. **Govern(統治)**：サイバーセキュリティリスク管理の戦略/方針/役割と責任を確立・監視します(CSF 2.0で追加)。
2. **Identify(識別)**：資産/リスク/体制を把握します。
3. **Protect(防御)**：予防/保護策(アクセス制御等)を実施します。
4. **Detect(検知)**：インシデント兆候を検出します。
5. **Respond(対応)**：事象発生後の対応/分析を行います。
6. **Recover(復旧)**：復旧と継続改善を行います。

### 実施階層(Tiers)

| Tier | 名称 | 特徴 |
|:-----|:-----|:-----|
| 1 | Partial | 部署ごとにバラバラ |
| 2 | Risk Informed | 部分的に統一 |
| 3 | Repeatable | 組織全体で標準化 |
| 4 | Adaptive | 継続改善しながら運用 |

### CSFの目的

セキュリティ対策を合理的に体系化する枠組みです。

- サイバーリスク管理を改善します。
- 組織全体のセキュリティ対策の習熟度を高めます。
- ISMSや各種規格と整合性をもって運用できます。

## サイバーセキュリティ経営ガイドライン

経営者が主体となってサイバーセキュリティ対策を推進するための指針です。
本ガイドラインは、以下の機関により策定・公表されています。

- 経済産業省(METI)
- 独立行政法人情報処理推進機構(IPA)

### 目的

IT部門任せにしない、経営課題としてのサイバーセキュリティを明確にします。

### サイバーセキュリティ経営ガイドラインの3原則

ガイドラインの中核となるのが以下3つの原則です。

#### 原則1：経営者の責任明確化

サイバーセキュリティ対策は、情報システム部門やセキュリティ担当者だけの責任ではなく、最終責任は経営者にあることを明確にしています。

また、経営者は以下を担います。

- 法人の決定
- 投資判断
- 組織体制の整備

#### 原則2：リスクマネジメントの実践

サイバーセキュリティは「発生確率 × 影響度」で評価すべき経営リスクです。

そのため、次を継続的に実施する必要があります。

- 重要情報資産の把握
- 脅威/脆弱性の把握
- リスク評価と優先順位付け

#### 原則3：サプライチェーン全体への配慮

自社だけではなく、委託先・子会社・取引先を含めたサプライチェーン全体のセキュリティを考慮します。

近年の標的型攻撃では、取引先経由の侵入や委託業者を踏み台にした攻撃が多いため、極めて重要な原則です。

### サイバーセキュリティ経営ガイドラインの重要10項目

3原則を具体化したものが重要10項目です。
これらは経営者が確認・支持すべき事項として整理されています。

#### 【重要項目1】責任者が責任を持つ体制の構築

- セキュリティ責任者(CISO等)の任命
- 権限と責任の明確化

#### 【重要項目2】サイバーセキュリティリスクの把握

- 重要情報資産の洗い出し
- 事業影響を考慮したリスク評価

#### 【重要項目3】リスク対応計画の策定

- 回避
- 低減
- 移転
- 受容

#### 【重要項目4】社内規定/ルールの整備

下記を整備し、組織に周知します。

- 情報セキュリティポリシー
- インシデント対応規程

#### 【重要項目5】従業員への教育/訓練

- フィッシング対策
- 標的型攻撃訓練
- 情報持ち出し禁止

**人に起因する事故対策が重要です。**

#### 【重要項目6】技術的対策の導入と運用

- ファイアウォール
- EDR
- パッチ管理

**技術は目的ではなく手段である点が重要です。**

#### 【重要項目7】インシデント対応体制の整備

- CSIRTの設置
- 初動対応手順の明確化

#### 【重要項目8】被害拡大防止と復旧計画

- BCP
- DR(災害復旧)

#### 【重要項目9】外部との情報共有

- IPA
- JPCERT/CC
- 業界団体

#### 【重要項目10】継続的改善(PDCA)

- 定期的な見直し
- 監査結果の反映

## クラウドサービス利用のための情報セキュリティマネジメントガイドライン

クラウドサービス利用に伴い増大するリスクに対し、**利用者が適切なセキュリティ管理を行うための指針** を提供するものです。

### 目的

1. クラウド利用に関するリスクの可視化
2. 利用者が実施すべき情報セキュリティ対策の提示
3. 利用者とクラウド事業者の **責任分界の明確化**

クラウド特有の依存関係を考慮したガイドラインであり、特に **責任共有モデルが中心的テーマ** となります。

### ガイドラインの構成

1. クラウドサービスの定義
2. セキュリティ上の留意点
3. 利用者が実施すべき管理策
4. 契約時に確認すべき事項
5. 外部委託管理(SLAの重要性)

### クラウド利用のリスクと特徴

クラウドは利便性が高い一方、以下の **クラウド固有リスク** が存在します。

#### データ管理場所の不透明性

データが国内外どこで保管されるか利用者が把握できないことがあります。

#### マルチテナント環境

他組織と同じ基盤を共有するため、設定ミスが重大な情報漏えいにつながります。

#### 可用性依存

障害が起きた場合、利用者側で復旧できず事業者に依存します。

#### サービス変更の影響

クラウド事業者はサービス仕様を随時変更するため、利用者の運用に影響が及びます。

#### データ消去の不確実性

退去時にデータが完全に消去されているか検証が難しいです。

### 利用者が実施すべき情報セキュリティ管理策

クラウド利用者には以下の管理策が求められます。

#### リスク分析の実施

- データ分類(機密性/可用性要件)
- クラウド依存による業務影響評価
- 復旧要件(RTO/RPO)の設定

#### 契約/SLAの確認

クラウド利用において最重要要素です。

**SLAで確認すべき項目：**

- データの所在国
- 可用性(例：99.9%)
- 事故時の連絡体制
- データ消去手順
- 監査証跡(ログ)の取得/提供
- サブプロセッサ(再委託先)の管理

#### アクセス管理

- 多要素認証
- アカウントの最小権限設定
- APIキーの保護

#### 暗号化管理

- 通信と保存データの暗号化
- 鍵管理の責任分界(利用者 or 事業者)

#### 運用監視/ログ管理

- アクセスログの取得/保管
- 異常動作の検知
- 事業者提供ログ＋自社ログの統合分析

### 契約時のチェックポイント

クラウド利用者が確認すべき項目は以下のとおりです。

#### セキュリティ対策の内容

ファイアウォール、IDS/IPS、バックアップ、暗号化などを明文化します。

#### 事故発生時の責任分担

責任共有モデルに基づき、「利用者責任」「事業者責任」を明確化します。

#### データの取扱い

- データ所有権が利用者にあることの確認
- 廃止時のデータ消去方法
- データ移行の手順

#### 監査権限

利用者が事業者へ **監査要求できるか** どうかは極めて重要です。
多くのクラウドでは、「SOC2報告書」「ISO/IEC 27001 証明書」「クラウドセキュリティ監査レポート」などで代替されます。

### 外部委託管理の重要性

クラウド事業者利用は **外部委託の一形態** であり、委託先管理が強く求められます。
以下が一般的な委託管理策です。

- 委託契約の明文化
- 委託先のセキュリティ基準(ISO 27001等)
- 再委託の管理
- 定期的なレビュー(サービス変更の確認)

### クラウド関連の国際標準(周辺知識)

#### ISO/IEC 27017

クラウド向けのセキュリティ管理策です。
利用者と事業者の「責任分界」が中心テーマです。

#### ISO/IEC 27018

パブリッククラウドにおける **個人情報保護(PII)管理** です。

#### NIST SP 800-144

クラウドセキュリティの総合的リスクガイドです。

#### Shared Responsibility Model(責任共有モデル)

AWS、Azure、GCP など全クラウドが採用しています。

- **事業者の責任：** 基盤/ハード/仮想化レイヤ
- **利用者の責任：** アプリ設定/ID管理/データ暗号化

## ソフトウェア管理ガイドライン

企業・団体がソフトウェアを合法的・安全に使用するための **組織的ルール・管理プロセスを体系化した指針** です。

### 目的

1. **ライセンス遵守(コンプライアンス)**
2. **脆弱性対策(パッチ管理)**
3. **違法コピー防止**
4. **セキュリティリスク低減**
5. **コスト最適化(資産管理)**

### 法人/団体が実施すべき基本事項

#### 管理体制の整備

- ソフトウェア管理責任者(SAM Manager)の明確化
- 管理ポリシー/運用手順の策定
- 管理対象(PC、サーバ、クラウド利用)の範囲を定義

#### ライセンスの取得/保有状況の把握

- 契約書/請求書/プロダクトキーなどの証跡保存
- **保有ライセンス数 >= 使用数** を維持
- EULA(エンドユーザライセンス契約)の遵守確認

#### インストール管理(変更管理)

- 無断インストールの禁止
- インストール担当者の限定
- アプリケーション配布サーバ/MDMなどによる統制

セキュリティ面では、不正ソフトインストール防止が重要です。

#### 使用状況の監査(インベントリ管理)

- 定期棚卸(半年/年次)
- インベントリツール(資産管理ツール)の使用
- 使用実績の分析(未使用ソフトは削除)

ISO/IEC 19770-2(SWIDタグ)で識別管理が提唱されています。

#### 保守管理(アップデート/脆弱性管理)

- パッチ適用プロセスの整備
- ベンダーのサポート期限確認
- 古いソフトの排除(Windows XP、Flash など)

#### 外部委託先のソフトウェア利用管理

- 委託先のライセンス遵守状況の確認
- SaaS利用時の利用規約(契約条件)の確認
- 再委託先管理

### ソフトウェア管理が必要な理由

#### 法的リスク(著作権法)

違法コピーや不正利用は **刑事罰/民事賠償** の対象となります。

#### セキュリティリスク

- マルウェア混在リスク
- 非正規アプリによる情報流出
- 脆弱性未修正のソフトが攻撃対象になります。

#### 経営リスク

- ライセンス監査による巨額請求
- シャドーITの増加による情報統制崩壊
- クラウド課金の肥大化

## 中小企業の情報セキュリティ対策ガイドラインと自己宣言

### 中小企業の情報セキュリティ対策ガイドラインとは

**独立行政法人 情報処理推進機構(IPA)** が策定した **中小企業が最低限実施すべき情報セキュリティ対策を示した実践的指針** です。

- 正式名称：「中小企業の情報セキュリティ対策ガイドライン」
- 策定主体：IPA(経済産業省所管)
- 最新版：Ver.3.1

### なぜ「中小企業向け」ガイドラインが必要なのか

#### 大企業向け規格の限界

ISO/IEC 27001 や NIST CSF は以下の理由で中小企業には負担が大きいです。

- 専任セキュリティ担当者がいません。
- コスト/人員/時間が限られています。
- 文書化/運用管理が過剰になりやすいです。

#### 実被害の増加

IPA の調査では、**サプライチェーン攻撃の踏み台として中小企業が狙われる事例が増加** しています。
そのため、「簡易/実践的/段階的」な指針が必要です。

### ガイドラインの基本構成

#### 3つのレベル構成

中小企業の成熟度に応じて段階的に対策できる構造です。

| レベル | 内容 |
|---|---|
| ★ | 最低限の対策(まず守る) |
| ★★ | 体制整備/運用の強化 |
| ★★★ | 組織的/継続的な改善 |

#### 対策分野

ガイドラインは以下の観点で整理されています。

- 組織的対策
- 人的対策
- 技術的対策
- 物理的対策

### 中小企業の自己宣言とは

**中小企業が、自社の情報セキュリティ対策状況を自己評価し、IPAが提供する仕組みを通じて対外的に宣言する制度です。**

正式名称：「SECURITY ACTION(セキュリティ対策自己宣言)」

#### 自己宣言の目的

- セキュリティ対策実施の「見える化」
- 取引先/顧客への信頼性向上
- サプライチェーン全体の安全性向上

### SECURITY ACTION の区分

#### 二段階の宣言レベル

| 区分 | 内容 |
|---|---|
| ★ 一つ星 | 情報セキュリティ5か条の実践 |
| ★★ 二つ星 | ガイドラインに基づく対策実施 |

#### 情報セキュリティ5か条

- OS/ソフトウェアを最新に保ちます。
- ウイルス対策ソフトを導入します。
- パスワードを適切に管理します。
- 重要情報を守ります。
- 問題発生時の対応を決めておきます。

**技術よりも「行動」に重点** が置かれています。

### よくある誤解

| 誤解 | 正解 |
|---|---|
| 自己宣言＝認証 | 認証ではありません |
| 専門家必須 | 自己評価が基本です |
| 技術対策中心 | 組織/人的対策が中心です |
| 大企業向け | 中小企業専用です |

## 共通フレーム2007(Common Frame 2007)

### 定義

ソフトウェア開発及び情報システム構築における業務内容/工程/成果物を共通の枠組みで整理・定義したガイドラインです。

- 策定主体：経済産業省
- 公表年：2007年
- 正式名称：共通フレーム2007（ソフトウェアライフサイクルプロセス）

### 制定の背景

従来のシステム開発では、下記のような課題がありました。

- 発注者と受注者で工程認識が異なります。
- 「どこまでが誰の責任か」が曖昧です。
- 契約トラブル・品質問題が多発していました。

共通フレーム2007は、発注者/受注者間が共通言語で議論できる土台を提供するために策定されました。

### 目的

1. 業務範囲/責任分担の明確化
2. 工程/成果物の共通認識形成
3. 契約/見積もり/進捗管理の透明化
4. 品質向上とトラブル防止

### 対象範囲

共通フレーム2007は、情報システムの企画から廃棄までの全ライフサイクルを対象とします。

```TEXT
企画 → 要件定義 → 設計 → 開発 → テスト → 運用・保守 → 廃棄
```

### 構成

#### プロセスの大分類

| 区分 | 内容 |
|---------|-------------|
| 企画プロセス | システム化構想・計画 |
| 要件定義プロセス | 業務要件・システム要件 |
| 開発プロセス | 設計・実装・テスト |
| 運用・保守プロセス | 運用管理・変更管理 |
| 支援プロセス | 品質管理・構成管理 |
| 組織的プロセス | 人材育成・改善活動 |

#### 特徴的なポイント

- 発注者側の作業も明示されています。
- 成果物が工程ごとに整理されています。
- 責任主体が分かりやすくなっています。

### 共通フレーム2007と他標準との関係

#### ISO/IEC 12207との関係

- ISO/IEC 12207：ソフトウェアライフサイクルプロセスの国際規格
- 共通フレーム2007：ISO/IEC 12207を日本向けに実務適用しやすく整理したもの

#### PMBOKとの違い

| 観点 | 共通フレーム2007 | PMBOK |
|------|------------|--------|
| 主眼 | 業務・工程・成果物 | プロジェクト管理 |
| 対象 | システム開発全体 | マネジメント |
| 利用者 | 発注者＋受注者 | 主に管理者 |

- 共通フレーム：何を作るか
- PMBOK：どう管理するか

### 共通フレーム2013への進化

#### 共通フレームのバージョン遷移

| 年 | 名称 | 位置づけ |
|------|----------------|----------------------|
| 2007年 | 共通フレーム2007 | 共通フレーム98の後継、ISO/IEC 12207:2002対応版 |
| 2013年 | **共通フレーム2013** | **現行最新版** |
| 2018年 | 共通フレーム2013(改訂) | 内容整理・補足(名称は2013のまま) |

#### 共通フレーム2007の限界

共通フレーム2007は優れた枠組みでしたが、以下の課題がありました。

- ウォーターフォール前提の色が強いです。
- クラウド/外部サービス利用を想定していません。
- 「開発中心」で「運用・サービス提供」が弱いです。

#### 共通フレーム2013での進化点

共通フレーム2013では、次の点が大きく拡張されています。

1. **「サービス提供」を明確に位置付け**
    - システムを作るだけではなく、「提供し続ける」ことを前提としています。
    - ITサービスマネジメント(ITIL)との親和性が向上しています。
2. **クラウド/アウトソーシング対応**
    - SaaS/PaaS/IaaS利用を前提としています。
    - 外部事業者(委託先)管理を明確化しています。
    - 契約/SLA/責任分界を重視しています。
3. **アジャイル/反復型開発も許容**
    - 特定の開発モデルを固定しません。
    - ウォーターフォール/アジャイル両対応です。

#### 共通フレーム2013の対象範囲

共通フレーム2007と同様に、ライフサイクル全体を対象としています。
ただし、「運用/サービス提供の比重増大」と「継続的改善」が明確に定義されています。

#### ITIL / ISO/IEC 20000 との関係

| 規格 | 主眼 |
|---------------|----------------|
| 共通フレーム2013 | 企画～運用までの全体枠組み |
| ITIL | サービス運用・改善 |
| ISO/IEC 20000 | サービスマネジメント要求事項 |

共通フレーム2013は上位の全体設計図です。

## 総務省/NICT と取組「NOTICE」

### 総務省(Ministry of Internal Affairs and Communications)

総務省は、日本の情報通信行政を担当しており、**サイバーセキュリティ確保/通信インフラ保護** を推進する行政機関です。

#### 役割

- 電気通信事業法に基づく通信サービスの監督
- インターネット基盤(IPアドレス/ドメイン管理)の整備
- サイバー攻撃対策(通信防護、インシデント対応支援)
- NICT等との研究開発/政策連携
- IoT機器の脆弱性対策(NOTICEの実施主体)

### NICT(国立研究開発法人 情報通信研究機構)

NICT は日本唯一の情報通信分野の公的研究機関です。
総務省の所管のもと、通信の安全性を確保するための技術開発を行います。

#### 主な役割

- サイバー攻撃観測網(ダークネット観測：NICTER)
- サイバー攻撃分析
- IoT機器の脆弱性スキャン(NOTICE 技術基盤の提供)
- インシデント対応技術の研究

特に **NICTER(ニクター)** は、日本最大級のサイバー攻撃観測網として有名です。

### NOTICE(National Operation Towards IoT Clean Environment)

NOTICE とは、**脆弱な IoT 機器を悪用したサイバー攻撃を未然に防ぐため、インターネット上からアクセス可能な機器の安全性を調査し、利用者へ注意喚起を行うプロジェクト** です。

#### 実施主体

- 総務省：制度設計/事業者への指導
- NICT：技術実施(スキャン/データ分析)
- ISP：利用者への通知(NOTICE=通知の意味)

### NOTICE の仕組み

1. NICT が **IoT 機器のポートスキャン** を実施します。
   - Telnet、SSH、HTTP、FTP など
   - 不正アクセスではなく **電気通信事業法に基づく正当な調査** です。
2. 脆弱性が疑われる機器(初期パスワード状態/公開設定など)を検出します。
3. NICT から ISP に情報提供します。
   - 個人情報は ISP が管理します。
   - NICT は個人識別不能な IP 情報のみ扱います。
4. ISP から利用者へ「注意喚起」を行います。
   - 初期パスワード変更
   - ファームウェア更新
   - 不要ポート封鎖

NOTICE は **利用者に直接連絡する唯一の国主導セキュリティ施策** です。

### NOTICE の法的根拠(電気通信事業法 164条の2)

総務省と NICT が通信をスキャンすることは、「通信の秘密」を損なうため本来禁止されます。
しかし NOTICE は電気通信事業法 第164条の2 により **特例として許可された制度** です。

目的は以下のとおりです。

- サイバー攻撃に悪用される IoT 機器を事前に把握します。
- 利用者の被害防止/踏み台化防止を図ります。

これは世界的にも先進的な枠組みです。

### NOTICE の背景：IoT 機器が狙われる理由

IoT の典型的な脆弱性は以下のとおりです。

- 初期パスワードが未変更
- Web管理画面が外部公開
- 古いファームウェア(脆弱性未修正)
- UPnP の誤設定
- Telnet/FTP などの平文プロトコルが開放

Mirai ボットネット(2016)以降、IoTは DDoS の踏み台として大量悪用されてきました。

### 関連する国のサイバー対策(周辺知識)

#### SOC(Security Operation Center)

官公庁向けには **政府機関SOC(NICTと内閣サイバーセンターが協力)** が存在します。

#### サイバー攻撃観測網：NICTER

- 重要インフラへの攻撃兆候を収集します。
- 企業や研究者にもデータ提供します。

#### NOTICE と混同されやすい施策

##### BOT対策プロジェクト(ISPによる自主的取組)

- DDoSの踏み台になっている PC を利用者へ通知します。
- こちらは ISP 主体で政府の法律による特例ではありません。

##### サイバークリーンセンター(旧施策)

- ボット駆除を目的とした政府事業です。
- 2011年3月に終了しています。

NOTICE はこれらとは目的も法的根拠も異なります。

## ISMS（情報セキュリティマネジメントシステム）

### ISMSの概要

ISMS（Information Security Management System）は、組織の情報セキュリティを体系的に管理するための仕組みです。
情報の **機密性（Confidentiality）・完全性（Integrity）・可用性（Availability）** の3要素（CIA）を維持・改善することを目的とします。

### ISO/IEC 27001（JIS Q 27001）

ISMSの **要求事項（Requirements）** を定めた国際規格です。
ISMS認証の基準となる規格であり、組織が「何をしなければならないか」を規定します。

#### 主要な構成

| 箇条 | 内容 |
|:-----|:-----|
| 4 | 組織の状況（利害関係者のニーズ、適用範囲） |
| 5 | リーダーシップ（経営者の責任、方針） |
| 6 | 計画（リスクアセスメント、リスク対応計画） |
| 7 | 支援（資源、力量、認識、コミュニケーション、文書化） |
| 8 | 運用（リスク対応の実施） |
| 9 | パフォーマンス評価（監視、内部監査、マネジメントレビュー） |
| 10 | 改善（不適合への対応、継続的改善） |

#### 附属書A（管理策）

ISO/IEC 27001の附属書Aには管理策の一覧が記載されています。
2022年版では **4テーマ93管理策** に再編されました（旧版は14分類114管理策）。

| テーマ | 管理策数 | 内容例 |
|:-------|:---------|:-------|
| 組織的管理策 | 37 | 情報セキュリティ方針、役割と責任 |
| 人的管理策 | 8 | 雇用前スクリーニング、教育訓練 |
| 物理的管理策 | 14 | 入退室管理、機器の保護 |
| 技術的管理策 | 34 | アクセス制御、暗号、ログ管理 |

#### 適用宣言書（SoA：Statement of Applicability）

組織が附属書Aの管理策から **採用する管理策と除外する管理策およびその理由** を記載した文書です。
ISMS認証審査において必須の文書であり、午後問題でも頻出です。

#### ISMS認証プロセス

```
申請 → 文書審査（Stage 1） → 実地審査（Stage 2） → 認証取得
  → 維持審査（年1回） → 更新審査（3年ごと）
```

- 認証機関：ISMS-AC（情報マネジメントシステム認定センター）が認定した審査機関
- 有効期間：3年間（維持審査を毎年実施）

### ISO/IEC 27002（JIS Q 27002）

ISMSの **管理策の実践のための規範（ベストプラクティス）** を定めた規格です。
27001の附属書Aの各管理策について、具体的な実施ガイダンスを提供します。

- 27001＝「何をすべきか」（要求事項）
- 27002＝「どうすべきか」（実践の手引き）

### ISMSのPDCAサイクル

ISMSは **PDCAサイクル** によって継続的に改善します。

| フェーズ | 内容 |
|:---------|:-----|
| Plan（計画） | ISMS方針策定、リスクアセスメント、リスク対応計画 |
| Do（実施） | 管理策の導入、教育訓練、運用 |
| Check（点検） | 監視・測定、内部監査、マネジメントレビュー |
| Act（改善） | 是正処置、予防処置、継続的改善 |

### ISMS関連規格の体系

| 規格 | 内容 |
|:-----|:-----|
| ISO/IEC 27000 | 用語集・概要 |
| ISO/IEC 27001 | ISMS要求事項（認証基準） |
| ISO/IEC 27002 | 管理策の実践規範 |
| ISO/IEC 27005 | 情報セキュリティリスクマネジメント |
| ISO/IEC 27017 | クラウドセキュリティ管理策 |
| ISO/IEC 27018 | クラウドにおけるPII保護 |
| ISO/IEC 27701 | プライバシー情報マネジメント（PIMS） |

### ISMSとNIST CSFの比較

| 観点 | ISMS（ISO 27001） | NIST CSF |
|:-----|:-------------------|:---------|
| アプローチ | 管理策ベース | リスクマネジメントベース |
| 認証制度 | あり（第三者認証） | なし（自己評価） |
| 対象 | 全業種・全規模 | 主に重要インフラ（汎用化） |
| 構造 | PDCAサイクル | 6機能（Govern/Identify/Protect/Detect/Respond/Recover） |
| 強制力 | 任意（ただし取引条件化する例あり） | 任意（米国では準拠を求める場合あり） |

## 情報セキュリティポリシー

### 情報セキュリティポリシーとは

組織の情報セキュリティに対する **基本的な考え方と方針を文書化したもの** です。
ISMSの中核文書であり、経営層の承認のもと策定・周知されます。

### 3階層構造

情報セキュリティポリシーは一般的に **3つの階層** で構成されます。

| 階層 | 名称 | 内容 | 策定者 |
|:-----|:-----|:-----|:-------|
| 第1層 | 基本方針（ポリシー） | 経営層の意思表明、目的・適用範囲・責任 | 経営層 |
| 第2層 | 対策基準（スタンダード） | 遵守すべき具体的な基準・ルール | 情報セキュリティ委員会 |
| 第3層 | 実施手順（プロシージャ） | 対策基準を実現するための具体的な手順書 | 各部門 |

#### 各階層の特徴

- **基本方針**：組織のセキュリティに対する姿勢を宣言するもので、変更頻度は低いです。
- **対策基準**：「パスワードは8文字以上」「USBメモリの使用禁止」など具体的なルールです。
- **実施手順**：「パスワード変更手順書」「インシデント報告手順」など運用マニュアルです。

#### 午後問題での出題ポイント

午後問題では、「情報セキュリティポリシーの見直し」「ポリシー違反への対処」がシナリオとして出題されます。
特に以下が重要です。

- 基本方針は **経営層（取締役会等）の承認** が必要です。
- 対策基準の策定には **リスクアセスメントの結果** を反映します。
- 実施手順は **定期的に見直し**、環境変化に対応します。
- 全従業員への **周知・教育** が不可欠です。

### 情報セキュリティ委員会

情報セキュリティポリシーの策定・運用を統括する組織横断的な委員会です。

| 役職 | 役割 |
|:-----|:-----|
| CISO（最高情報セキュリティ責任者） | 情報セキュリティの最終責任者 |
| 情報セキュリティ管理者 | 各部門のセキュリティ施策の推進 |
| 情報セキュリティ監査責任者 | 独立した立場で監査を実施 |

## リスクマネジメント

### リスクマネジメントの概要

リスクマネジメントは、組織の目的達成を阻害するリスクを **特定・分析・評価し、適切に対応する** 一連のプロセスです。
情報セキュリティにおいては、ISMSの中核プロセスとして位置づけられます。

### ISO 31000（リスクマネジメント規格）

リスクマネジメントの **国際規格** であり、あらゆる分野に適用できる汎用フレームワークです。

#### リスクマネジメントプロセス

```
コミュニケーション及び協議
  ↓
適用範囲・状況・基準の設定
  ↓
リスクアセスメント
  ├─ リスク特定
  ├─ リスク分析
  └─ リスク評価
  ↓
リスク対応
  ↓
モニタリング及びレビュー
  ↓
記録及び報告
```

### リスクアセスメント

リスクアセスメントは、以下の3つのサブプロセスで構成されます。

#### 1. リスク特定

情報資産に対する **脅威と脆弱性** を洗い出し、リスクを認識します。

- 情報資産の洗い出し（台帳作成）
- 脅威の特定（自然災害、サイバー攻撃、内部不正等）
- 脆弱性の特定（設定不備、パッチ未適用、教育不足等）

#### 2. リスク分析

特定されたリスクの **発生可能性と影響度** を分析します。

| 分析手法 | 内容 | 特徴 |
|:---------|:-----|:-----|
| 定量的リスク分析 | 金額で評価（ALE = SLE × ARO） | 客観的だが算出が困難 |
| 定性的リスク分析 | 高/中/低で評価（マトリクス） | 簡易だが主観的 |

**定量的リスク分析の用語：**

| 用語 | 意味 |
|:-----|:-----|
| ALE（年間予想損失額） | 年間に予想される損失の合計 |
| SLE（単一予想損失額） | 1回の事象で予想される損失額 |
| ARO（年間発生率） | 年間に事象が発生する確率 |
| AV（資産価値） | 情報資産の価値 |
| EF（暴露係数） | 資産が受ける損害の割合 |

**計算式：** ALE = SLE × ARO = AV × EF × ARO

#### 3. リスク評価

分析結果に基づき、リスクの **優先順位を決定** し、対応が必要なリスクを選定します。

- リスク基準（受容可能なリスクレベル）との比較
- リスクマトリクス（発生可能性 × 影響度）による優先順位付け

### リスク対応

リスク評価の結果に基づき、以下の **4つの戦略** からリスク対応を選択します。

| 戦略 | 内容 | 例 |
|:-----|:-----|:---|
| リスク回避 | リスクの原因となる活動を中止 | 危険なサービスの利用停止 |
| リスク低減（軽減） | 管理策の導入でリスクを下げる | ファイアウォール導入、暗号化 |
| リスク移転（共有） | リスクを第三者に移す | サイバー保険、外部委託 |
| リスク受容（保有） | リスクを認識した上で受け入れる | コストに見合わない場合 |

**残留リスク** ：リスク対応後も残るリスクであり、経営者が **残留リスクを承認** する必要があります。

### ISO/IEC 27005

情報セキュリティに特化したリスクマネジメント規格です。
ISO 31000の枠組みをISMSに適用する際のガイダンスを提供します。

## 情報セキュリティ監査

### 情報セキュリティ監査の概要

情報セキュリティ監査は、組織の情報セキュリティ対策が **適切に整備・運用されているかを独立した立場から検証** する活動です。

### 情報セキュリティ監査基準

経済産業省が策定した、情報セキュリティ監査を実施する **監査人が遵守すべき行為規範** です。

#### 監査の種類

| 種類 | 内容 | 目的 |
|:-----|:-----|:-----|
| 保証型監査 | 監査対象の情報セキュリティが基準に適合していることを **保証する** | 利害関係者（取引先等）への信頼性提供 |
| 助言型監査 | 改善すべき事項を **助言する** | 組織内部のセキュリティ改善 |

**午前問題では「保証型」と「助言型」の違いが頻出です。**

#### 監査の独立性

- 監査人は監査対象から **独立した立場** でなければなりません（外観上の独立性・精神上の独立性）。
- 内部監査であっても、監査対象部門の担当者が自部門を監査することは認められません。

### 情報セキュリティ管理基準

経済産業省が策定した、組織が情報セキュリティ管理において **実施すべき管理策のガイドライン** です。
ISO/IEC 27001/27002をベースに、日本の組織向けに整理されています。

- 監査基準＝「監査人の行動規範」（どう監査するか）
- 管理基準＝「管理策の基準」（何を管理するか）

### 監査プロセス

```
監査計画の策定 → 予備調査 → 本調査 → 監査報告書の作成 → フォローアップ
```

| フェーズ | 内容 |
|:---------|:-----|
| 監査計画 | 監査目的・範囲・スケジュール・体制の策定 |
| 予備調査 | 文書レビュー、組織体制の把握 |
| 本調査 | インタビュー、実地確認、証拠収集 |
| 監査報告 | 監査所見・改善勧告の報告 |
| フォローアップ | 改善状況の追跡確認 |

### 内部監査と外部監査

| 観点 | 内部監査 | 外部監査 |
|:-----|:---------|:---------|
| 実施者 | 組織内の監査部門 | 外部の監査法人・専門機関 |
| 目的 | 内部統制の改善 | 第三者による客観的評価 |
| 独立性 | 組織内での独立性 | 組織外からの完全な独立性 |
| ISMSでの位置づけ | 27001箇条9で必須 | 認証審査（任意） |

### システム監査との違い

| 観点 | 情報セキュリティ監査 | システム監査 |
|:-----|:---------------------|:-------------|
| 対象 | 情報セキュリティ管理体制 | 情報システム全般 |
| 基準 | 情報セキュリティ監査基準 | システム監査基準 |
| 観点 | 機密性・完全性・可用性 | 信頼性・安全性・効率性 |
| 策定 | 経済産業省 | 経済産業省 |

## ISO/IEC 15408（Common Criteria：コモンクライテリア）

### 概要

ISO/IEC 15408は、IT製品やシステムのセキュリティ機能を **客観的に評価するための国際規格** です。
日本では **JIS X 5070** として規格化されています。

通称 **CC（Common Criteria）** と呼ばれ、製品のセキュリティ品質を国際的に比較可能にします。

### CCの主要概念

| 用語 | 英語 | 意味 |
|:-----|:-----|:-----|
| TOE | Target of Evaluation | 評価対象（製品やシステム） |
| PP | Protection Profile | セキュリティ要件を定義した文書（利用者側が作成） |
| ST | Security Target | TOEのセキュリティ仕様書（開発者側が作成） |
| EAL | Evaluation Assurance Level | 評価保証レベル（1～7） |

#### PP（プロテクションプロファイル）とST（セキュリティターゲット）の関係

- **PP**：「こういうセキュリティ機能が欲しい」という **要求仕様**（利用者・調達者が定義）
- **ST**：「この製品はこういうセキュリティ機能を持つ」という **設計仕様**（開発者が定義）
- STはPPに適合することを示す必要があります。

### EAL（評価保証レベル）

| レベル | 名称 | 内容 |
|:-------|:-----|:-----|
| EAL1 | 機能テスト | 最低限の評価 |
| EAL2 | 構造テスト | 開発文書の確認 |
| EAL3 | 方式テスト・チェック | 開発環境の確認 |
| EAL4 | 方式設計・テスト・レビュー | 一般的な商用製品の評価レベル |
| EAL5 | 準形式的設計・テスト | 高セキュリティ製品 |
| EAL6 | 準形式的検証済み設計・テスト | 高リスク環境向け |
| EAL7 | 形式的検証済み設計・テスト | 最高レベル（軍事等） |

**EAL4が商用製品で最も一般的な評価レベル** です。

### CCの認証制度

- **CCRA（Common Criteria Recognition Arrangement）**：CC認証の相互承認協定
- 加盟国間でCC認証結果を相互に認め合います。
- 日本は **JISEC（ITセキュリティ評価及び認証制度）** を運用しています。
  - 認証機関：IPA
  - 評価機関：IPAが認定した民間評価機関

### 午前問題での出題ポイント

- PP/ST/TOEの定義と関係
- EALのレベル概要（特にEAL4）
- CCRAによる相互承認

## CRYPTREC（暗号技術検討会）

### 概要

CRYPTREC（Cryptography Research and Evaluation Committees）は、電子政府で使用する暗号技術の **安全性を評価・監視する** プロジェクトです。

- 運営：総務省、経済産業省、NICT、IPA
- 目的：電子政府における暗号技術の安全性確保

### 電子政府推奨暗号リスト

CRYPTRECが策定する暗号技術のリストは **3つのカテゴリ** で構成されます。

| カテゴリ | 内容 | 例 |
|:---------|:-----|:---|
| 電子政府推奨暗号リスト | 安全性・実装性に優れ、推奨される暗号 | AES、RSA、SHA-256、ECDSA |
| 推奨候補暗号リスト | 将来推奨暗号リスト入りが見込まれる暗号 | Camellia（一部用途） |
| 運用監視暗号リスト | 互換性維持のため使用を認めるが、推奨しない暗号 | SHA-1、3-key Triple DES |

#### 試験で問われるポイント

- **SHA-1は運用監視暗号リスト** に移行しています（推奨ではない）。
- **AES、SHA-256、RSA（2048bit以上）** は電子政府推奨暗号です。
- 暗号技術の危殆化（きたいか）により、リストは定期的に見直されます。

### 暗号の危殆化

暗号アルゴリズムの安全性が **計算能力の向上や解読手法の発見により低下する** ことです。

- 鍵長の短い暗号（DES、RSA 1024bit）は危殆化しています。
- 量子コンピュータの実用化により、RSA・ECDSAの安全性が将来脅かされる可能性があります（耐量子暗号の研究が進行中）。

## 政府機関等のサイバーセキュリティ対策のための統一基準群

### 概要

**内閣サイバーセキュリティセンター（NISC）** が策定する、政府機関および独立行政法人等が遵守すべきセキュリティ対策の基準です。

### 構成

| 文書 | 内容 |
|:-----|:-----|
| 統一基準群の概要 | 基準群全体の目的・構成の説明 |
| 政府機関等のサイバーセキュリティ対策のための統一規範 | 各府省庁が遵守すべき上位方針 |
| 政府機関等のサイバーセキュリティ対策のための統一基準 | 具体的な対策基準（遵守事項） |
| 政府機関等の対策基準策定のためのガイドライン | 対策基準の解説・実装ガイド |

### NISCの役割

- **NISC（National center of Incident readiness and Strategy for Cybersecurity）**
- 内閣官房に設置されたサイバーセキュリティの司令塔
- サイバーセキュリティ基本法に基づく **サイバーセキュリティ戦略本部** の事務局

#### 主な業務

- 政府機関のセキュリティ監査・監視
- サイバーセキュリティ戦略の策定
- 重要インフラ防護の調整
- インシデント対応の支援・調整（GSOC：Government Security Operation Coordination team）

## CSIRT（Computer Security Incident Response Team）

### CSIRTの概要

CSIRTは、コンピュータセキュリティインシデントに対応するための **専門チーム** です。
インシデントの検知・分析・対応・復旧を担い、組織のセキュリティを維持します。

### CSIRTの種類

| 種類 | 対象 | 例 |
|:-----|:-----|:---|
| 組織内CSIRT | 自組織のインシデント対応 | 企業内CSIRT |
| 国際連携CSIRT | 国レベルのインシデント調整 | JPCERT/CC |
| コーディネーションセンター | 複数CSIRTの連携調整 | FIRST |
| ベンダCSIRT | 自社製品の脆弱性対応 | PSIRT（Product SIRT） |
| セクターCSIRT | 業界固有のインシデント対応 | 金融ISAC、ICT-ISAC |

### CSIRTの主要機能

| 機能 | 内容 |
|:-----|:-----|
| インシデントハンドリング | インシデントの受付・トリアージ・対応・報告 |
| 脆弱性ハンドリング | 脆弱性情報の収集・分析・対応調整 |
| 状況把握 | 脅威情報の収集・分析（脅威インテリジェンス） |
| 普及啓発 | セキュリティ教育・注意喚起 |
| 技術支援 | フォレンジック・マルウェア解析等 |

### インシデントハンドリングの流れ

```
検知・受付 → トリアージ（優先度判定） → インシデントレスポンス
  → 報告・情報共有 → 事後対応（再発防止）
```

### JPCERT/CC

- **一般社団法人 JPCERT コーディネーションセンター**
- 日本における **インターネット定点観測、脆弱性情報のハンドリング、インシデント対応の調整** を行う組織
- 政府機関ではなく **民間の中立組織**
- 国際的なCSIRTコミュニティ **FIRST** のメンバー

#### 主な活動

- インシデント報告の受付・対応調整
- 脆弱性情報の収集・公開（JVN：Japan Vulnerability Notes）
- 注意喚起・セキュリティアラートの発信
- 制御システムセキュリティ（ICS-CERT）

### 日本シーサート協議会（NCA）

日本国内の **CSIRT間の情報共有と連携** を促進する組織です。
加盟組織同士が信頼関係に基づきインシデント情報を共有します。

## SOC（Security Operation Center）

### SOCの概要

SOCは、組織のネットワークやシステムを **24時間365日監視** し、セキュリティイベントの検知・分析を行う組織です。

### CSIRTとSOCの違い

| 観点 | SOC | CSIRT |
|:-----|:----|:------|
| 主な役割 | 監視・検知（事前） | 対応・復旧（事後） |
| 活動時間 | 24時間365日の常時監視 | インシデント発生時に活動 |
| 使用ツール | SIEM、IDS/IPS、EDR | フォレンジック、マルウェア解析 |
| 位置づけ | 「目」（見つける） | 「手」（対処する） |

### SOCの監視対象

- ファイアウォール/IDS/IPSのログ
- サーバ・ネットワーク機器のログ
- EDR/アンチウイルスのアラート
- SIEM（Security Information and Event Management）による統合分析

### SIEM

SIEMは、複数のセキュリティ機器やサーバから **ログを収集・統合・相関分析** するツールです。

- ログの一元管理
- リアルタイムのイベント相関分析
- アラートの自動生成
- コンプライアンス対応のレポート生成

## PCI DSS（Payment Card Industry Data Security Standard）

### 概要

PCI DSSは、クレジットカード情報を取り扱う事業者が遵守すべき **国際的なセキュリティ基準** です。
国際カードブランド5社（Visa、Mastercard、JCB、American Express、Discover）が共同で策定しました。

### 12の要件

| 要件番号 | 内容 |
|:---------|:-----|
| 1 | ネットワークセキュリティの管理策の導入・維持 |
| 2 | すべてのシステムコンポーネントへのセキュアな設定の適用 |
| 3 | 保存されたアカウントデータの保護 |
| 4 | オープンな公共ネットワークでの送信時の暗号化 |
| 5 | 悪意のあるソフトウェアからの保護 |
| 6 | セキュアなシステムおよびソフトウェアの開発・維持 |
| 7 | 業務上の必要性に基づくアクセス制限 |
| 8 | ユーザの識別とアクセスの認証 |
| 9 | カード会員データへの物理的アクセスの制限 |
| 10 | システムコンポーネントへのアクセスのログ記録・監視 |
| 11 | システムおよびネットワークのセキュリティの定期的テスト |
| 12 | 情報セキュリティポリシーの維持 |

### 準拠の評価方法

| 方法 | 内容 | 対象 |
|:-----|:-----|:-----|
| QSA（認定セキュリティ評価機関）による審査 | 外部監査 | 大規模加盟店・サービスプロバイダ |
| SAQ（自己問診） | 自己評価 | 中小規模加盟店 |
| ASV（認定スキャニングベンダ）によるスキャン | 外部脆弱性スキャン | 外部接続を持つ事業者 |

### PCI DSSの最新動向

PCI DSS v4.0（2022年3月公開）では以下が強化されています。

- カスタマイズアプローチの導入（柔軟な準拠方法）
- 多要素認証の要件強化
- リスクベースの脆弱性管理

## IPAセキュリティ関連ガイドライン

### 安全なウェブサイトの作り方

IPAが公開する **Webアプリケーションの脆弱性対策ガイドライン** です。
午後問題でWebアプリケーションセキュリティが出題される際の基本知識となります。

#### 対象とする主な脆弱性

| 脆弱性 | 対策の概要 |
|:-------|:---------|
| SQLインジェクション | プレースホルダの使用、エスケープ処理 |
| クロスサイトスクリプティング（XSS） | 出力時のエスケープ、Content-Security-Policy |
| CSRF（クロスサイトリクエストフォージェリ） | トークンの使用、Referer検証 |
| OSコマンドインジェクション | シェルコマンドの使用回避 |
| HTTPヘッダインジェクション | 改行コードの排除 |
| セッション管理の不備 | セッションIDの推測困難化、再生成 |
| ディレクトリトラバーサル | パス名の検証、固定ディレクトリ外アクセスの禁止 |

### 脆弱性届出制度

IPAが運営する **ソフトウェア製品やWebサイトの脆弱性を届け出る制度** です。

- 根拠法令：経済産業省告示「ソフトウェア等脆弱性関連情報取扱基準」
- 届出先：IPA
- 調整機関：JPCERT/CC
- 公開：JVN（Japan Vulnerability Notes）

#### 届出の流れ

```
発見者 → IPA（届出受付）→ JPCERT/CC（調整）→ 開発者（修正）→ JVN（公開）
```

### OWASP（Open Worldwide Application Security Project）

Webアプリケーションセキュリティに特化した **オープンなコミュニティプロジェクト** です。

#### OWASP Top 10（2021年版）

| 順位 | カテゴリ |
|:-----|:---------|
| A01 | アクセス制御の不備 |
| A02 | 暗号化の失敗 |
| A03 | インジェクション |
| A04 | 安全でない設計 |
| A05 | セキュリティの設定ミス |
| A06 | 脆弱で古いコンポーネント |
| A07 | 識別と認証の失敗 |
| A08 | ソフトウェアとデータの整合性の不具合 |
| A09 | セキュリティログとモニタリングの失敗 |
| A10 | SSRF（サーバサイドリクエストフォージェリ） |

- 契約管理
- インベントリ管理
- 更新管理
- コスト最適化
- コンプライアンス管理

### シャドーIT

ユーザが未許可のソフトやクラウドサービスを利用する行為です。

リスクは以下のとおりです。

- 情報漏えい
- 設定不備による脆弱性
- 組織の監査証跡が残りません。

### パッチ管理(Patch Management)

最重要のセキュリティ統制です。

流れは以下のとおりです。

1. 脆弱性情報の収集
2. 影響評価(CVSSなど)
3. パッチ適用テスト
4. 本番反映
5. 証跡/ログ管理

### サプライチェーンセキュリティ

- 取引先の脆弱性が自社リスクになります。
- 中小企業は攻撃の「踏み台」にされやすいです。

自己宣言は **取引条件の信頼材料** になります。
