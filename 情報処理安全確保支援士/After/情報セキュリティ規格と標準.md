## ISO/IEC 15408(Common Criteria：CC)

**情報システムや製品のセキュリティ機能が「どの程度信頼できるか」を評価するための国際規格です。**

### 主旨

- 製品(OS、FW、暗号モジュール等)が備えるセキュリティ機能を **客観的に評価** します。
- 認証されたシステムは **国際相互承認(CCRA)** により海外でも通用します。

### CC の三構成要素

| 要素 | 内容 |
|------|------|
| **PP(Protection Profile)** | 製品カテゴリごとの安全要求(例：ファイアウォール PP) |
| **ST(Security Target)** | 個別製品ごとに定義された安全要求 |
| **TOE(Target of Evaluation)** | 評価対象そのもの |

### EAL(Evaluation Assurance Level)

評価保証レベル 1～7 が定められています。
EAL が高いほど「検証の深度」が高くなります(設計検証/テスト/形式手法など)。

試験で問われやすいポイントは以下のとおりです。

- **EAL は「信頼性の評価レベル」であり、製品のセキュリティ性能そのものを保証するわけではありません。**

## ISO/IEC 27001(情報セキュリティマネジメントシステム：ISMS)

**組織が情報セキュリティを体系的に管理するための仕組み(ISMS)の要求事項を定めた国際規格です。**
ISMS認証の基準となる最も重要な規格であり、日本では **JIS Q 27001** として規格化されています。

### 目的

組織の情報資産について **機密性(Confidentiality)・完全性(Integrity)・可用性(Availability)** を維持し、利害関係者の信頼を確保することです。

### 規格の構成(箇条4～10)

ISO/IEC 27001は **Annex SL(ISOマネジメントシステム共通の高位構造)** に準拠しています。

| 箇条 | 内容 | 試験での要点 |
|:-----|:-----|:-------------|
| 4 | 組織の状況 | 適用範囲の決定、利害関係者の特定 |
| 5 | リーダーシップ | 経営者のコミットメント、情報セキュリティ方針 |
| 6 | 計画 | **リスクアセスメント**、リスク対応計画、セキュリティ目的 |
| 7 | 支援 | 資源、力量、認識、コミュニケーション、**文書化した情報** |
| 8 | 運用 | リスク対応の実施、運用計画と管理 |
| 9 | パフォーマンス評価 | 監視・測定、**内部監査**、**マネジメントレビュー** |
| 10 | 改善 | 不適合への是正処置、**継続的改善** |

### 附属書A(Annex A)の管理策

ISO/IEC 27001:2022では、附属書Aの管理策が **4テーマ93管理策** に再編されました(旧2013年版は14分類114管理策)。

| テーマ | 管理策数 | 対象例 |
|:-------|:---------|:-------|
| 組織的管理策 | 37 | 方針、役割と責任、脅威インテリジェンス |
| 人的管理策 | 8 | 雇用前スクリーニング、教育訓練 |
| 物理的管理策 | 14 | 入退室管理、機器保護、クリアデスク |
| 技術的管理策 | 34 | アクセス制御、暗号、ログ管理、DLP |

### 適用宣言書(SoA：Statement of Applicability)

附属書Aの管理策から **採用する管理策と除外する管理策およびその理由** を記載した文書です。
ISMS認証審査において **必須の文書** であり、午後問題での頻出テーマです。

### ISMS認証プロセス

```
申請 → 文書審査(Stage 1) → 実地審査(Stage 2) → 認証取得
  → 維持審査(年1回) → 更新審査(3年ごと)
```

| 項目 | 内容 |
|:-----|:-----|
| 認定機関 | ISMS-AC(情報マネジメントシステム認定センター) |
| 審査機関 | ISMS-ACが認定した審査登録機関 |
| 有効期間 | 3年間(毎年の維持審査が必要) |

### ISMSのPDCAサイクル

| フェーズ | 内容 |
|:---------|:-----|
| Plan(計画) | ISMS方針策定、リスクアセスメント、リスク対応計画 |
| Do(実施) | 管理策の導入・運用、教育訓練 |
| Check(点検) | 監視・測定、内部監査、マネジメントレビュー |
| Act(改善) | 是正処置、継続的改善 |

### 27001と27002の関係

| 観点 | ISO/IEC 27001 | ISO/IEC 27002 |
|:-----|:--------------|:--------------|
| 性格 | 要求事項(shall) | 実践の手引き(should) |
| 役割 | 「何をすべきか」を規定 | 「どうすべきか」を解説 |
| 認証 | 認証対象 | 認証対象外 |
| 附属書A | 管理策一覧(概要) | 各管理策の詳細ガイダンス |

### 試験ポイント

- 27001は **認証可能な要求事項規格** であり、27002は **ベストプラクティス集** です。
- 午後問題では「ISMSの導入・運用」「リスクアセスメント」「適用宣言書」が頻出です。
- **経営者のコミットメント** が不可欠であることが繰り返し問われます。

## ISO/IEC 27002(情報セキュリティ管理策の実践規範)

**ISO/IEC 27001 における「付属書Aの管理策」を具体化したガイドライン規格です。**

### 目的

組織が導入すべき情報セキュリティ管理策を体系的に提供します。
リスクに基づく情報セキュリティ管理策の適用を支援し、組織が情報セキュリティ水準を確立・維持するための手引きとなります。
ISMS(ISO/IEC 27001)における Annex A 管理策の詳細説明としても位置づけられています。

### 管理策ドメイン構成(2022版)

- 組織的管理策(Organizational)
- 人的管理策(People)
- 物理的管理策(Physical)
- 技術的管理策(Technological)

### 代表的な管理策

- 物理的入退室管理
- バックアップ
- アクセス制御
- 暗号管理
- ログ監視
- セキュリティインシデント管理
- サプライチェーンセキュリティ

### 「雇用」に関する管理策の位置づけ(ISO/IEC 27002:2022)

雇用に関する情報セキュリティ管理策は、従来の「第7章 Human Resource Security(人的セキュリティ)」に相当する領域であり、以下の3つのフェーズに分かれます。

1. **雇用前(Before Employment)**
2. **雇用期間中(During Employment)**
3. **雇用終了/異動時(Termination or Change of Employment)**

#### 雇用前のコントロール(採用段階)

情報漏えい・不正アクセス・内部不正を未然に防ぐため、採用の段階からセキュリティ確保が求められます。

##### 背景調査(Screening)

**ISO/IEC 27002 管理策 6.1(旧7.1.1)**

採用候補者に対して、以下を実施して適性を確認します。

- 身元確認(Identity Check)
- 資格および経歴の検証
- 犯罪歴の確認(法令が許す範囲で)
- 前職での不正行為の有無

目的は、悪意ある内部者(Insider Threat)の排除や、不正アクセス・情報持ち出しの防止です。
試験で問われやすいポイントとして、採用時点から情報セキュリティ対策が開始されるという点があります。

##### 雇用条件へのセキュリティ要件組込み

労働契約にセキュリティ遵守事項を明記します。

記載すべき項目は以下のとおりです。

- 機密保持義務(NDA)
- 情報資産の取り扱いルール
- 会社支給デバイスの使用条件
- ログ監視の実施(プライバシーとの整合性)

#### 雇用期間中のコントロール

##### セキュリティ教育/訓練(Awareness Training)

従業員はセキュリティ知識が乏しいと脅威になります。
教育は「継続的」に実施することが必須です。

- パスワード管理
- フィッシング/ソーシャルエンジニアリング対策
- マルウェア対策
- 設備の利用ルール
- 情報の分類と取り扱い方法

##### 責務の明確化

従業員がどの情報資産にアクセスできるか、どの責任を持つかを明確化します。

- 担当業務に必要最小限のアクセス権のみ付与(Least Privilege)
- 職務分掌の明確化(例：購入と承認を別担当が行う)

##### ログ監視と行動監査

内部不正は意図的・故意であるため、技術的制御(アクセス制御/ログ/監視)が極めて重要です。

#### 雇用終了/異動時のコントロール

##### アクセス権の即時削除

手続きの遅れは **内部不正の最大リスク要因** となります。
退職日/異動日と同時に以下を実施します。

- アカウント無効化
- 物理鍵(ICカード)の回収
- VPN/クラウド/AD権限の削除

##### 資産返却

返却確認はチェックリストで実施します。

- PC
- USB
- 入退室カード
- 機密文書

##### 退職後も秘密保持義務は継続

労働契約やNDA(機密保持契約)により、退職後も機密情報の漏洩は禁止されます。

#### 雇用に関する管理策の要点まとめ

| 区分 | キーワード | 試験での狙われ方 |
|------|-------------|---------------------|
| 雇用前 | 背景調査、契約書への記載 | 採用時点から対策開始 |
| 雇用中 | 教育、職務分掌、最小権限 | 人的脆弱性の低減 |
| 雇用終了 | アカウント削除、資産返却 | 退職者リスクの防止 |

### 試験ポイント

27002 は「情報セキュリティ管理策の網羅的ガイドライン」であり、**実装方法の統一ではなく「ベストプラクティスの参照」に使う規格です。**

## ISO/IEC 27017(クラウドセキュリティ管理策)

**クラウドサービス(IaaS/PaaS/SaaS)に特化したセキュリティ管理策です。**
27002 をベースに、クラウド固有の問題点を補完します。

### 目的

クラウドの **利用者(Cloud Customer)** とクラウド **サービス提供者(CSP)** の双方に必要なセキュリティ管理策を提供します。

### 代表的クラウド特有管理策

| 管理策 | 内容 |
|--------|------|
| **責任共有モデル** | 利用者と CSP のセキュリティ責任範囲を明確化 |
| **仮想化管理策** | ハイパーバイザ保護、VM分離 |
| **管理者アクセスの保護** | API 鍵管理、多要素認証 |
| **顧客環境の分離(Multi-tenancy)** | テナント間干渉防止 |
| **ログの連携/取得範囲** | CSP が提供するログの明確化 |

### 試験ポイント

- 27002 との違いは **クラウド固有リスクに対応しているかどうか** です。
- 特に SaaS/IaaS で「誰が何を守るのか」が重要です。

## ISO/IEC 27018(パブリッククラウドにおけるPII保護)

**パブリッククラウド環境で個人識別情報(PII：Personally Identifiable Information)を処理する際の管理策を定めた国際規格です。**
27002の管理策をベースに、クラウドにおけるプライバシー保護の観点から追加・拡張されています。

### 目的

- パブリッククラウドにおける **PIIプロセッサ(クラウド事業者)** の義務を明確化します。
- PIIの取扱いに関する **透明性と信頼性** を確保します。
- GDPR等のプライバシー法令への準拠を支援します。

### 主な要求事項

| 項目 | 内容 |
|:-----|:-----|
| PII処理目的の制限 | 顧客が指定した目的以外でPIIを処理しない |
| PII返却・消去 | 契約終了時のPII返却・完全消去の保証 |
| PIIの移転制限 | PIIの国外移転に関する通知と同意 |
| 透明性の確保 | PIIの処理場所・再委託先の開示 |
| PIIの開示要求対応 | 法執行機関からの開示要求への対応手順 |
| データ侵害通知 | PII侵害時の顧客への速やかな通知 |

### 27017と27018の使い分け

| 観点 | ISO/IEC 27017 | ISO/IEC 27018 |
|:-----|:--------------|:--------------|
| 焦点 | クラウド全般のセキュリティ | クラウドにおけるプライバシー(PII) |
| 対象者 | 利用者＋提供者の双方 | 主にPIIプロセッサ(提供者) |
| 補完対象 | ISO/IEC 27002 | ISO/IEC 27002 + プライバシー法 |

### 試験ポイント

- 27018は **クラウドにおけるPII保護に特化** した規格です。
- GDPRや個人情報保護法との関連で出題されることがあります。
- 27017(セキュリティ全般)と27018(プライバシー)の違いを識別する力が求められます。

## ISO/IEC 27014(情報セキュリティガバナンス)

**経営層が組織の情報セキュリティをどのように統治(ガバナンス)すべきかを定めた国際規格です。**

### 目的

- 情報セキュリティに対する **経営層の責任と関与** を明確化します。
- 情報セキュリティを **組織全体のガバナンス** に統合します。
- セキュリティ活動と **事業目的の整合性** を確保します。

### ガバナンスの6つの原則

| 原則 | 内容 |
|:-----|:-----|
| 組織全体のセキュリティ確立 | 情報セキュリティが組織全体をカバーすること |
| リスクベースのアプローチ | ビジネスリスクに基づく意思決定 |
| 投資判断の方向性 | セキュリティ投資の妥当性確保 |
| 内外の要件との適合 | 法令・規制・契約への準拠 |
| セキュリティに適した環境 | セキュリティ文化の醸成 |
| パフォーマンスレビュー | 事業成果に対するセキュリティの貢献度評価 |

### ガバナンスとマネジメントの違い

| 観点 | ガバナンス(27014) | マネジメント(27001) |
|:-----|:-------------------|:--------------------|
| 主体 | 経営層(取締役会等) | 管理者(CISO・情報セキュリティ委員会) |
| 目的 | 方向づけ・監視・評価 | 計画・実施・運用・改善 |
| 視点 | 戦略的・全社的 | 戦術的・実務的 |
| 関係 | マネジメントを統治する | ガバナンスの下で実行する |

### 試験ポイント

- **ガバナンス(統治)とマネジメント(管理)の違い** は午前問題で頻出です。
- 経営層の情報セキュリティへの関与・責任が繰り返し問われます。

## ISO/IEC 30147(IoT セキュリティ開発ライフサイクル)

**IoT 製品やサービスのセキュリティを、開発ライフサイクル全体で確保するための国際規格です。**

### 対象

- センサー
- IoT デバイス
- ゲートウェイ
- クラウド IoT サービス

### 主な要求事項

| 開発プロセス | 内容 |
|---------------|--------|
| **脅威分析(Threat Modeling)** | STRIDE に類似、攻撃経路を洗い出す |
| **セキュア設計** | 認証/更新/暗号化を前提とした設計 |
| **セキュア実装** | ソースコードの安全性、脆弱性排除 |
| **テストと評価** | ペネトレーションテスト、更新テスト |
| **セキュアアップデート** | OTA 更新の署名検証必須 |
| **廃棄フェーズ対応** | データ消去、鍵削除 |

### 特徴

- IoT 製品向けの「セキュア開発プロセス」規格です。
- 27017 がクラウド、30147 が IoT を補完する関係にあります。

## JIS Q 27000:2014(情報セキュリティマネジメントシステムの用語規格)

**JIS Q 27000:2014** は ISO/IEC 27000:2014 と整合した **情報セキュリティマネジメントシステム(ISMS)に関する用語集規格** です。
情報セキュリティ分野で使用される専門用語(資産/脅威/脆弱性/リスク/管理策など)の定義を標準化し、ISMS(ISO/IEC 27001)の運用/監査における共通理解(共通言語)を提供しています。

### 目的

- ISMS に関する用語の意味を統一します。
- セキュリティ管理/リスク分析の前提を明確化します。
- 企業間/国際間でのコミュニケーションを円滑化します。

### 情報セキュリティ(Information Security)

情報の **機密性(Confidentiality)/完全性(Integrity)/可用性(Availability)** を維持することです。

- **機密性**：許可された者だけがアクセスできます。
- **完全性**：正確で、改ざんされていません。
- **可用性**：必要なときに利用できます。

### 情報セキュリティリスク(Information Security Risk)

**脅威が脆弱性を悪用することにより、資産に対して生じる可能性のある結果(影響)** です。

式で表現すると下記のとおりです。

```
リスク ＝ 脅威 × 脆弱性 × 資産価値
```

### リスク構成要素の詳細

#### 資産(Asset)

組織にとって価値のあるものです。
例：情報、業務システム、データ、機器、人などがあります。

#### 脅威(Threat)

資産に損害を与える可能性のある事象です。
例：攻撃者による不正アクセス、マルウェア、災害、内部不正などがあります。

#### 脆弱性(Vulnerability)

脅威が成立する原因となる弱点です。
例：設定不備、パッチ未適用、アクセス制御不足、設計上の欠陥などがあります。

### リスクの分類

- **投機的リスク**：相場などのように利益と損失のどちらかが生じる可能性のある不確実性のことです。
- **純粋リスク**：損失のみが生じる可能性のある不確実性のことです。

### 情報セキュリティリスクアセスメント(ISO/IEC 27005 に基づく)

#### Step 1：資産の識別

情報資産、システム、サービスを洗い出します。

#### Step 2：脅威/脆弱性の識別

資産を脅かす要因を整理し、脆弱性を把握します。

#### Step 3：リスク分析

影響度 × 発生可能性 で評価します(定性/定量いずれも可)。

#### Step 4：リスク評価

許容可能か、対応が必要かを判断します。

#### Step 5：リスク対応

- **回避(Avoid)**：リスク源をなくします。
- **低減(Mitigate)**：セキュリティ対策を講じます。
- **移転(Transfer)**：保険加入や外部委託を行います。
- **受容(Accept)**：残余リスクを受け入れます。

### 管理策(ISO/IEC 27001 A.5～A.18)

ISMS ではこれらを組み合わせてリスクを低減します。

#### 技術的管理策

- アクセス制御、IDS/IPS、暗号化、ログ管理

#### 組織的管理策

- 情報セキュリティポリシ、職務分掌、教育訓練

#### 物理的管理策

- 入退室管理、監視カメラ、サーバ室管理

## JIS X 9401:2016(クラウドコンピューティングの概要と用語)

**JIS X 9401:2016(ISO/IEC 17788:2014 と一致)** は、クラウドコンピューティングに関する **概要と用語の標準規格** です。

### 規格の目的

- クラウドに関する用語の定義統一
- 各サービスモデル(SaaS/PaaS/IaaS)の役割整理
- クラウドエコシステム(利用者/提供者/仲介者)の明確化
- クラウド管理/ガバナンスの基本概念提供

### クラウドサービスモデル(JIS X 9401/ISO 17788 準拠)

#### SaaS(Software as a Service)

利用者は **アプリケーションをそのまま利用** するサービスモデルです。

- インフラ/プラットフォーム/アプリケーションすべてをプロバイダが管理します。
- 利用者は設定やデータ入力のみに集中します。
- 最も利用者の管理負担が少ないモデルです。

例：Gmail、Microsoft 365、Salesforce、会計ソフト、CRM、グループウェア

利用者の責任：アカウント管理、データ保護、アクセス制御設定

#### PaaS(Platform as a Service)

アプリケーションを構築する **実行基盤(OS/ミドルウェア)を提供** するサービスです。

- 開発者は **環境構築不要でアプリ開発に集中** できます。
- OS、ランタイム、DB 等は提供者が管理します。
- スケールアウト機能なども提供されます。

例：Google App Engine、Microsoft Azure App Service、AWS Elastic Beanstalk

利用者の責任：アプリケーションコード、データ構造/設定、API 利用

#### IaaS(Infrastructure as a Service)

**仮想サーバ、ネットワーク、ストレージなど基盤そのもの** を提供します。

- OS/ミドルウェアは利用者が管理します。
- 柔軟性が高く、オンプレと最も近いモデルです。
- セキュリティ設定の責任が大きくなります。

例：Amazon EC2、Microsoft Azure VM、Google Compute Engine

利用者の責任：OS/パッチ管理、ミドルウェア設定、ファイアウォール設定、監視/ログ管理

### 3サービスモデルの比較(責任共有モデル)

**IaaS → PaaS → SaaS の順で利用者負担が軽減** します。
IPA試験では特に **責任範囲の違い** が問われます。

| 項目 | SaaS | PaaS | IaaS |
|------|------|------|------|
| アプリ管理 | × | △(コードのみ) | ○ |
| ミドルウェア | × | ×(提供側) | ○ |
| OS管理 | × | × | ○ |
| 仮想化層 | × | × | ×(提供側) |
| ネットワーク | × | × | × |
| データ管理 | 利用者 | 利用者 | 利用者 |

### クラウドの特性(JIS X 9401/ISO 17788)

クラウドは次の5特性を満たします(NIST SP 800-145 にも一致)。

1. **オンデマンド自動サービス(On-demand self-service)**
2. **広範囲ネットワークアクセス(Broad network access)**
3. **リソース共有(Resource pooling)**
4. **迅速な弾力性(Rapid Elasticity)**
5. **計測サービス(Measured service)**

これらにより、従来のオンプレミスとは異なる高度なスケール/可用性を実現します。

### クラウドの導入におけるリスク(支援士頻出)

#### データ管理リスク

- データの所在(海外リージョン)
- 法規制(GDPR、個人情報保護法)

#### 可用性リスク

- 障害時のクラウド依存
- 冗長構成の不足

#### セキュリティリスク

- 設定ミス(S3 バケット公開)
- IAM権限不備
- API 鍵漏えい

#### 外部委託管理リスク

- SLAの不備
- サードパーティ(再委託)の確認不足

## JIS Q 15001(個人情報保護マネジメントシステム：PMS)

**組織が個人情報を適切に管理するためのマネジメントシステムの要求事項を定めた日本独自の規格です。**
プライバシーマーク(Pマーク)認定の基準となります。

### プライバシーマーク(Pマーク)制度

| 項目 | 内容 |
|:-----|:-----|
| 運営機関 | JIPDEC(日本情報経済社会推進協会) |
| 認定基準 | JIS Q 15001 |
| 有効期間 | 2年間(更新審査あり) |
| 対象 | 国内に事業所を持つ事業者 |
| マーク | 「P」マークの使用権が付与される |

### JIS Q 15001 と ISO/IEC 27001 の違い

| 観点 | JIS Q 15001(Pマーク) | ISO/IEC 27001(ISMS) |
|:-----|:---------------------|:--------------------|
| 保護対象 | **個人情報** に特化 | 情報資産全般(機密性・完全性・可用性) |
| 規格種別 | 日本独自のJIS規格 | 国際規格(ISO) |
| 認証範囲 | 事業者単位(法人全体) | 適用範囲を自由に設定可能 |
| 法令との関係 | 個人情報保護法と密接に連携 | 法令を含むリスク全般 |
| 認定機関 | JIPDEC | ISMS-AC認定の審査機関 |

### 試験ポイント

- Pマークは **個人情報保護に特化** した認定制度であり、ISMSとは保護対象が異なります。
- 午前問題では「PマークとISMSの違い」として出題されます。
- JIS Q 15001は **日本国内でのみ通用** する規格(国際的な相互認証なし)です。

## ISO 22301(事業継続マネジメントシステム：BCMS)

**組織が事業中断のリスクに備え、事業継続能力を確保するためのマネジメントシステム規格です。**
日本では **JIS Q 22301** として規格化されています。

### 目的

- 災害・サイバー攻撃・パンデミック等による **事業中断からの復旧能力** を組織に確保します。
- 事業継続に関する計画(BCP)の策定・維持・改善を体系的に行います。

### 主要概念

| 用語 | 意味 |
|:-----|:-----|
| BCP(事業継続計画) | 事業中断時の対応手順・復旧計画を文書化したもの |
| BIA(事業影響度分析) | 事業中断による影響を評価し、優先業務を特定する分析 |
| RTO(目標復旧時間) | 事業を復旧させるまでの目標時間 |
| RPO(目標復旧時点) | データをどの時点まで復旧するかの目標 |
| MTPD(最大許容停止時間) | 事業が停止を許容できる最大時間 |

### BIA(事業影響度分析)の流れ

```
重要業務の特定 → 影響度の評価 → RTO/RPO/MTPDの設定 → 必要資源の特定
```

### BCMSのPDCA

| フェーズ | 内容 |
|:---------|:-----|
| Plan | BIA実施、リスクアセスメント、BCP策定 |
| Do | BCP運用、教育訓練、演習(テスト) |
| Check | 演習結果のレビュー、内部監査 |
| Act | BCPの見直し・改善 |

### 試験ポイント

- **BIAによる重要業務の特定** と **RTO/RPOの設定** が午後問題で頻出です。
- BCPは「策定して終わり」ではなく、**定期的な演習と見直し** が必要です。
- サイバー攻撃(ランサムウェア等)を想定したBCPの出題が増加しています。

## FIPS 140-2/140-3(暗号モジュールのセキュリティ要件)

**NIST(米国国立標準技術研究所)が策定した、暗号モジュールが満たすべきセキュリティ要件を定めた連邦政府標準規格です。**

### 概要

- FIPS 140-2(2001年)：広く普及した暗号モジュール検証標準
- FIPS 140-3(2019年)：ISO/IEC 19790に整合した後継規格
- 検証プログラム：**CMVP(Cryptographic Module Validation Program)**

### セキュリティレベル

| レベル | 内容 | 用途例 |
|:-------|:-----|:-------|
| Level 1 | 基本的なセキュリティ要件(ソフトウェア暗号) | 一般的なソフトウェア製品 |
| Level 2 | 物理的なタンパー証跡(改ざん検知) | 商用暗号装置 |
| Level 3 | 物理的なタンパー耐性(改ざん防止) | 高セキュリティ環境 |
| Level 4 | 環境攻撃への完全な耐性 | 軍事・政府機関向け |

### 試験ポイント

- ISO/IEC 15408(CC)が **IT製品全般のセキュリティ評価** であるのに対し、FIPS 140は **暗号モジュールに特化** した検証規格です。
- Level 1～4の概要を理解しておくことが重要です。
- 暗号製品の調達要件として問われることがあります。

## ISO/IEC 20000-1(サービスマネジメントシステム)

### JIS Q 20000-1:2020 とは

JIS Q 20000-1 は以下を規定する **第三者認証可能なマネジメントシステム規格** です。

- サービス提供者が満たすべき **サービスマネジメントシステム(SMS)の要求事項**
- 顧客へ一定品質のサービスを提供するための仕組み
- ITIL などのベストプラクティスを **制度として実装** する枠組み

ITIL は **参考となる実践書(ベストプラクティス)** であり、JIS Q 20000-1 は **組織に義務として課すマネジメントシステム要求事項** である点が重要な違いです(試験頻出)。

### サービスマネジメントシステム(SMS)とは

SMS(Service Management System)は **サービス提供を計画/運用/評価/改善するための組織的仕組み** を指します。
ISO マネジメントシステムの構造である **Annex SL(高位構造)** に基づき、以下の 7 要素で構成されます。

#### 組織の状況(Context of the Organization)

- 組織の目的、利害関係者、ニーズ/期待の把握
- 適用範囲(サービス範囲)の明確化
- 内部/外部の課題の特定

#### リーダーシップ(Leadership)

IT サービスを提供する責任者の明確化が必須です。

- トップマネジメントによるコミットメント
- **方針(Service Management Policy)** の制定
- SMS の役割と責任の割り当て

#### 計画(Planning)

- リスク及び機会への取り組み
- SMS 目標の設定
- サービス管理プロセスを含む計画立案

#### 支援(Support)

- **資源の確保(人/技術/財務)**
- 文書化した情報の管理
- 力量/教育
- コミュニケーション

#### 運用(Operation)

ここが SMS の中心であり、JIS Q 20000-1 が規定する **サービス管理プロセス群の実装** が求められます。

#### パフォーマンス評価(Performance Evaluation)

サービスの品質が維持されているかを継続的に評価します。

- モニタリング(KPI)
- 内部監査
- マネジメントレビュー

#### 改善(Improvement)

- 不適合への是正処置
- システムの継続的改善

### JIS Q 20000-1 が求めるサービス管理プロセス

JIS Q 20000-1:2020 は具体的な **サービス管理プロセス** を要求事項として明示しています。

#### サービス提供プロセス群

- **サービスレベル管理(SLA 管理)**
- サービスレポート
- サービス継続及び可用性管理
- 予算/会計管理(コスト管理)
- 容量/性能管理
- 情報セキュリティ管理
- 供給者管理(サプライヤ管理)

#### サービス運用プロセス群

- インシデント管理
- サービス要求管理
- 問題管理(Problem Management)
- 構成管理(CMDB)
- 変更管理(Change Control)
- リリース管理

これらは ITIL と対応していますが、**JIS Q 20000 は「管理すべきこと」を記述し、方法論には踏み込まない** ことが特徴です。

### 継続的改善(PDCA)

継続的改善は SMS の最重要要素です。
ISO マネジメントシステム共通の **PDCA サイクル** を基本とします。

#### Plan(計画)

- サービス目標の設定
- リスクアセスメント
- サービス設計

#### Do(実行)

- サービス管理プロセスの運用
- インシデント対応
- 変更管理/リリース

#### Check(評価)

- KPI 評価
- 内部監査
- サプライヤ評価
- マネジメントレビュー

#### Act(改善)

- 是正処置(原因分析 → 再発防止)
- サービスの最適化
- プロセス改善提案

継続的改善は単なる「問題対応」ではなく、**SMS 全体の有効性向上** を目的とする点が重要です。

## 周辺知識

### ISMS(Information Security Management System)

ISO/IEC 27001 に基づく組織の情報セキュリティ管理の仕組みです。
ISMS 運用は **Plan → Do → Check → Act** で継続的改善します。

### SoA(適用宣言書)

採用/不採用の管理策と理由を明確化します。

### リスク受容基準(Risk Acceptance Criteria)

リスクを許容可能と判断するための基準です。

### ITIL との違い(頻出)

| 項目 | ISO/IEC 20000(JIS) | ITIL |
|------|-----------------------|------|
| 性質 | 規格(義務/審査可能) | ベストプラクティス(参考書) |
| 焦点 | 要求事項/SMS 構築 | 方法論/プロセス手順 |
| 認証 | 第三者認証可能 | 認証不可(個人資格は除く) |

### SLA の重要性

- サービスレベルを明確化します。
- 監視対象(KPI)を定義します。
- 契約の基準となります(可用性/応答時間など)。

### サプライヤ管理(外部委託)

クラウド、AMS(運用委託)など他組織と協働する場合、**責任分界点の明確化(Shared Responsibility Model)** が必須です。

### クラウドサービス提供者/利用者の役割

JIS X 9401 ではエコシステムとして明確に定義されています。

### マルチテナンシ(Multi-tenancy)

複数利用者が同一物理基盤を共有する仕組みです。

### API セキュリティ

IaaS/PaaS では API 認証が特に重要です。

### SOC報告書(SOC 1/SOC 2/SOC 3)

SOC(Service Organization Control)報告書は、**外部委託先(クラウド事業者等)の内部統制を第三者が評価した報告書** です。
AICPA(米国公認会計士協会)が策定した基準に基づき、監査法人等が作成します。

| 種類 | 対象 | 内容 | 利用者 |
|:-----|:-----|:-----|:-------|
| SOC 1 | 財務報告に係る内部統制 | 委託先の業務処理統制の有効性 | 経営者・監査法人 |
| SOC 2 | セキュリティ等の統制 | Trust Service Criteria(5原則)に基づく評価 | 顧客・関係者(NDA下) |
| SOC 3 | SOC 2の簡易版 | 概要レベルの報告 | 一般公開可能 |

**SOC 2のTrust Service Criteria(信頼性サービス基準)：**

| 基準 | 内容 |
|:-----|:-----|
| Security(セキュリティ) | 不正アクセスからの保護(必須) |
| Availability(可用性) | サービスの可用性確保 |
| Processing Integrity(処理の完全性) | 処理の正確性・適時性 |
| Confidentiality(機密性) | 機密情報の保護 |
| Privacy(プライバシー) | 個人情報の保護 |

**Type I** と **Type II** の違い：

| Type | 評価内容 |
|:-----|:---------|
| Type I | ある時点での統制の設計の適切性 |
| Type II | 一定期間にわたる統制の運用の有効性 |

クラウドサービスの委託先管理(ISO/IEC 27001の箇条8やサイバーセキュリティ経営ガイドラインの重要10項目)において、**SOC 2 Type II報告書** の入手・確認が重要です。

### 内部不正(Insider Threat)

人的セキュリティ領域で最大リスクです。

#### 内部不正の典型例

- 持ち出し(USB/私物PC)
- 顧客情報の売却
- クラウドの不正設定変更
- 自作アカウントによるバックドア作成

### 職務分掌(Separation of Duties)

不正を成立させないための組織的統制です。
例：支払申請者と承認者を分離します。

### 最小権限の原則(Least Privilege)

必要最小限の権限しか付与しないことで不正に耐性を持たせる原則です。
IAM(Identity & Access Management)の重要概念です。

### アカウントライフサイクル管理

アカウントは以下のライフサイクルで管理します。
どこかで遅れると **不正アクセスに悪用されます**。

1. 発行
2. 変更
3. 権限見直し
4. 退職時削除(即時)

## 試験で問われる各規格の識別ポイント

試験で問われる観点は以下のとおりです。

- **ISO/IEC 15408**：セキュリティ機能の評価規格(EAL)です。
- **ISO/IEC 27001**：ISMS の要求事項(認証基準)です。
- **ISO/IEC 27002**：情報セキュリティ管理策(ガイドライン)です。
- **ISO/IEC 27014**：情報セキュリティガバナンス(経営層の統治)です。
- **ISO/IEC 27017**：クラウド向けの管理策です。
- **ISO/IEC 27018**：クラウドにおけるPII(個人情報)保護です。
- **ISO/IEC 30147**：IoT のセキュア開発工程です。
- **ISO 22301**：事業継続マネジメントシステム(BCMS)です。
- **JIS Q 15001**：個人情報保護マネジメントシステム(Pマーク基準)です。
- **JIS Q 27000**：ISMS の用語規格です。
- **JIS X 9401**：クラウドコンピューティングの概要と用語です。
- **ISO/IEC 20000-1**：サービスマネジメントシステムの要求事項です。
- **FIPS 140-2/140-3**：暗号モジュールのセキュリティ検証です。

それぞれの **焦点がどこにあるかを明確に識別する力** が求められます。
