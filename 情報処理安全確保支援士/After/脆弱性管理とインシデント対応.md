## CVSS v3(Common Vulnerability Scoring System v3)

### CVSS v3 とは

**CVSS(Common Vulnerability Scoring System)** とは、脆弱性の深刻度を共通の基準で数値化(0.0～10.0)するための評価体系です。

#### 公式仕様

- FIRST(Forum of Incident Response and Security Teams)
- CVSS v3.0/v3.1 Specification

### CVSS を使う目的

| 観点 | 目的 |
|---|---|
| 技術 | 脆弱性の危険度を客観評価 |
| 運用 | 対応優先度の判断 |
| 経営 | リスク説明/報告 |

**CVSSは「攻撃可能性 × 影響度」を数値で示す指標** であり、**脆弱性そのものの深刻度** を表します。

### CVSS v3 の全体構造

CVSS v3 は **3つの評価基準** から構成されます。

```
CVSS v3
├─ 基本評価基準(Base)
├─ 現状評価基準(Temporal)
└─ 環境評価基準(Environmental)
```

**Base → Temporal → Environmental の順に現実に近づきます。**

### 基本評価基準(Base Metrics)

#### 位置づけ(最重要)

**基本評価基準** は、以下の特徴を持ちます。

- 脆弱性そのものの性質を表します。
- 時間/環境に依存しません。

**全ての CVSS スコアの土台** であり、**CVE 公開時に必ず算出されます。**

#### 基本評価基準の構成

基本評価基準は **2系統** からなります。

| 系統 | 内容 |
|---|---|
| 攻撃容易性(Exploitability) | 攻撃のしやすさ |
| 影響度(Impact) | 被害の大きさ |

#### 攻撃容易性の評価項目

| 指標 | 意味 |
|---|---|
| AV(Attack Vector) | 攻撃経路(Network/Local等) |
| AC(Attack Complexity) | 攻撃の難易度 |
| PR(Privileges Required) | 必要な権限 |
| UI(User Interaction) | 利用者操作の要否 |

#### 影響度の評価項目

| 指標 | 意味 |
|---|---|
| C(Confidentiality) | 機密性への影響 |
| I(Integrity) | 完全性への影響 |
| A(Availability) | 可用性への影響 |

### 現状評価基準(Temporal Metrics)

**現状評価基準** は、脆弱性を取り巻く「現在の状況」を反映して基本評価基準を補正する評価です。

#### 評価項目

| 指標 | 内容 |
|---|---|
| E(Exploit Code Maturity) | 攻撃コードの成熟度 |
| RL(Remediation Level) | 対策の提供状況 |
| RC(Report Confidence) | 情報の信頼度 |

#### 現状評価基準の特徴

- 時間とともに変化します。
- パッチ公開でスコアが下がる場合があります。
- **Base × Temporal = 現実的リスク** です。

**「攻撃コードが出回っているか」** が鍵となります。

### 環境評価基準(Environmental Metrics)

**環境評価基準** は、脆弱性が存在する **自組織のシステム環境/重要度** を考慮して調整する評価です。

#### 主な評価要素

| 観点 | 内容 |
|---|---|
| 資産重要度 | 機密性/完全性/可用性の重み付け |
| 修正影響 | 自組織固有の影響 |
| 攻撃範囲 | 実際の配置環境 |

#### 環境評価基準の本質

- 組織ごとに値が異なります。
- 経営判断/優先度決定に直結します。
- **SOC/CSIRT が最も重視する指標** です。

### 3つの評価基準の違い(試験頻出)

| 評価基準 | 何を表すか | いつ使うか |
|---|---|---|
| 基本評価 | 脆弱性そのもの | 公開時 |
| 現状評価 | 攻撃/対策の状況 | 運用判断 |
| 環境評価 | 自組織への影響 | 対応優先度 |

### CVSS v3 と v2 の違い

| 項目 | v2 | v3 |
|---|---|---|
| 権限概念 | 曖昧 | PRで明確化 |
| 影響範囲 | 単純 | Scope概念追加 |
| 精度 | 低 | 高 |

## CVSS v4.0

### CVSS v4.0 とは

2023年11月にFIRSTが公開したCVSSの最新バージョンです。v3.1からの大幅な改訂により、より正確な脆弱性評価が可能になりました。

### CVSS v4.0 の評価基準の構成

v3の3つの評価基準から、v4.0では**4つのメトリクスグループ**に再編されました。

| メトリクスグループ | 内容 | v3との対応 |
|------------------|------|-----------|
| **Base Metrics** | 脆弱性固有の特性 | Base Metrics相当 |
| **Threat Metrics** | 脅威の現状（旧Temporal） | Temporal Metrics相当（簡素化） |
| **Environmental Metrics** | 自組織環境への影響 | Environmental Metrics相当 |
| **Supplemental Metrics** | 補足情報（スコアに影響しない） | **新規追加** |

### v3.1 から v4.0 への主な変更点

| 変更点 | v3.1 | v4.0 |
|--------|------|------|
| Attack Requirements | なし | **AT（攻撃要件）が新設**：攻撃成立に必要な前提条件 |
| Scope | Scope(S)で影響範囲を表現 | Scopeを廃止し、**脆弱なシステム/後続システムごとにCIA影響を分離** |
| User Interaction | None/Required | None/Passive/**Active**（3段階に細分化） |
| Temporal → Threat | E/RL/RCの3指標 | **Exploit Maturity(E)のみ**に簡素化 |
| 命名規則 | CVSS:3.1/AV:N/... | **CVSS:4.0/AV:N/...**（バージョン明記） |

### Supplemental Metrics（補足メトリクス）

スコア算出には影響しないが、リスク判断の補助情報として活用されます。

| 指標 | 内容 |
|------|------|
| **Safety(S)** | 人命への影響（ICS/IoT向け） |
| **Automatable(AU)** | 攻撃の自動化可能性 |
| **Recovery(R)** | システムの回復可能性 |
| **Value Density(V)** | 攻撃成功時に得られる価値の密度 |
| **Vulnerability Response Effort(RE)** | 対応に要する労力 |
| **Provider Urgency(U)** | ベンダが示す緊急度 |

### CVSS v4.0 のスコア体系

v4.0ではメトリクスの組み合わせにより、4種類のスコアが算出されます。

| スコア名 | 使用メトリクス | 用途 |
|---------|-------------|------|
| **CVSS-B** | Base のみ | 基本的な深刻度評価 |
| **CVSS-BT** | Base + Threat | 脅威状況を加味した評価 |
| **CVSS-BE** | Base + Environmental | 環境を加味した評価 |
| **CVSS-BTE** | Base + Threat + Environmental | **最も現実に即した評価** |

### 試験対策ポイント

- v4.0では**Scope概念が廃止**され、脆弱なシステムと後続システムの影響を分離して評価します
- **Temporal MetricsがThreat Metricsに改名**され、Exploit Maturityのみに簡素化されました
- **Supplemental Metrics**はスコアに影響しない補足情報です
- v3.1とv4.0の**違いを問う午前問題**が出題される可能性があります

## JVN(Japan Vulnerability Notes)と脆弱性対策ポータル

### JVN(Japan Vulnerability Notes)

**JVN(Japan Vulnerability Notes)** は、日本国内で利用されるソフトウェア等の脆弱性情報を提供するポータルサイトです。

#### 運営主体

JVN は以下の 2 組織が共同運営しています。

- IPA(独立行政法人 情報処理推進機構)
- JPCERT/CC(一般社団法人 JPCERT コーディネーションセンター)

#### JVN の役割

- 脆弱性情報の公開(国内ソフトウェア中心)
- 対応状況(パッチ/緩和策/回避策)の掲載
- 開発者と調整(調整済み情報を公開)
- CVE ID との対応づけ
- CWE、CVSS による評価の提示

#### 特徴

- **日本語で利用可能** です。
- 国内ソフトウェア/IoT 機器など **日本で普及している製品に関する情報が充実** しています。
- 漏えい事故を防ぐため、開発者との調整期間後に公開されます。

### JVN iPedia(脆弱性対策データベース)

**JVN iPedia(脆弱性対策データベース)** は、JVN と密接に連携しつつ、国内外の脆弱性情報を幅広く集約するデータベースです。

#### 運営主体

- IPA(独立行政法人 情報処理推進機構)

#### 情報源

JVN iPedia は以下から脆弱性情報を収集し、データベース化します。

- JVN
- NVD(National Vulnerability Database：米国)
- ベンダーのセキュリティ情報
- 海外 CERT の公表情報

#### 特徴

- **日本語で脆弱性情報を検索可能** です。
- 製品カテゴリ別の統計情報を提供します。
- マルチベンダ製品の CVE 対応状況を確認できます。
- CVSS v3 に基づく脆弱性深刻度を提供することで、ユーザ/企業のリスク評価に役立てます。

## CWE(Common Weakness Enumeration)

**CWE(Common Weakness Enumeration)** は、MITRE 社が管理する **「ソフトウェアの脆弱性につながる根本原因(弱点)」を分類・体系化した標準リスト** です。

### CWE が扱うもの

CWE は **脆弱性(CVE)そのものではありません**。
扱うのは下記のような脆弱性の根本原因です。

- CWE-79: クロスサイトスクリプティング(XSS)
- CWE-89: SQL インジェクション
- CWE-22: ディレクトリトラバーサル
- CWE-352: CSRF
- CWE-787: Out-of-bounds Write

### 目的

- 脆弱性の原因を体系的に理解します。
- セキュア開発/レビューの品質向上に役立てます。
- 開発者/研究者向けの共通語彙(共通フレームワーク)として利用します。

### CWE と CVE の違い

| 項目 | CWE | CVE |
|------|------|------|
| 対象 | 脆弱性を生む「弱点」 | 実際に発見された脆弱性 |
| 例 | 「入力検証不足」 | 「Log4j の RCE 脆弱性」 |
| 管理 | MITRE | MITRE |
| 役割 | 原因の分類 | 個々の脆弱性識別子 |

### JVN/JVN iPedia/CVE/CWE の関係図

```
[脆弱性の原因] ──→ CWE(弱点分類)
       |
       ↓(実際の脆弱性として発見)
     CVE(固有ID)
       |
       ↓
 NVD/JVN/JVN iPedia で公開
```

## 脆弱性ハンドリングと届出制度

### 情報セキュリティ早期警戒パートナーシップ

経済産業省の告示に基づき、**脆弱性情報の届出から公表までを調整する日本独自の枠組み**です。

#### 制度の法的根拠

- 経済産業省告示「ソフトウエア等脆弱性関連情報取扱基準」（2004年施行）
- 情報処理の促進に関する法律

#### 3つの関係者と役割

| 関係者 | 組織 | 役割 |
|--------|------|------|
| **受付機関** | IPA（独立行政法人 情報処理推進機構） | 脆弱性情報の届出を受け付けます |
| **調整機関** | JPCERT/CC | 製品開発者との調整を行い、公表日を調整します |
| **製品開発者** | ベンダ各社 | 脆弱性の修正・パッチの提供を行います |

#### 届出から公表までのフロー

```
発見者 → IPAに届出(受付機関)
          ↓ 届出情報を通知
        JPCERT/CC(調整機関)
          ↓ 脆弱性情報を通知・対応調整
        製品開発者(パッチ開発)
          ↓ 対策完了
        JVN で脆弱性情報を公表（調整済み公表）
```

#### 届出の2つの対象

| 対象 | 説明 | 届出先 |
|------|------|--------|
| **ソフトウェア製品の脆弱性** | OS、アプリ、ライブラリ等の脆弱性 | IPAが受付→JPCERT/CCが調整→JVNで公表 |
| **ウェブサイトの脆弱性** | 特定のウェブサイトの脆弱性 | IPAが受付→ウェブサイト運営者に通知 |

- ウェブサイトの脆弱性はJVNでは公表されず、**運営者への通知のみ**です
- 発見者は**正当な方法**で発見した脆弱性のみ届出できます（不正アクセスによる発見は不可）

### 脆弱性の責任ある開示（Coordinated Vulnerability Disclosure）

| 開示方式 | 内容 | リスク |
|---------|------|--------|
| **協調的な開示(CVD)** | ベンダと調整後にパッチと同時に公表します | 低（推奨される方式） |
| **完全開示(Full Disclosure)** | 発見後すぐに全情報を公開します | 高（ゼロデイ攻撃の恐れ） |
| **非開示** | 発見者が脆弱性を秘匿します | 中（発見者のみが悪用可能） |

- 日本の早期警戒パートナーシップは**CVD方式**に基づいています
- 国際的にはISO/IEC 29147（脆弱性開示）、ISO/IEC 30111（脆弱性ハンドリング）が標準です

### ゼロデイ脆弱性

- パッチや回避策が提供される**前に**攻撃に悪用される脆弱性です
- 発見から対策までの期間が「0日」であることから命名されました
- **ゼロデイ攻撃**：ゼロデイ脆弱性を利用した攻撃です
- 対策：WAF/IPSによる暫定的防御、ベンダへの早期報告、脅威インテリジェンスの活用

## SCAP(Security Content Automation Protocol)

ITシステムの脆弱性、設定状態、ソフトウェアの識別情報などを標準化された形式で表現・交換・評価するための仕様群です。

### SCAPが必要とされた背景

従来のセキュリティ運用には以下の課題がありました。
これを解決するために、NISTが中心となって策定した共通言語群が SCAP です。

- ベンダごとに脆弱性IDや表記が異なります。
- 手作業による脆弱性管理で属人化します。
- 構成チェック結果がツール間で共有できません。

### SCAPの目的

SCAPの目的は次の3点に集約できます。

1. 脆弱性・設定情報の標準化
2. 自動評価・自動監査の実現
3. ツール間の相互運用性確保

### SCAPの全体構造

SCAPは単一規格ではなく、複数の標準仕様の集合体であり、これを「SCAPの構成要素」と呼びます。

#### CVE(Common Vulnerabilities and Exposures)

脆弱性を一意に識別するための共通ID体系です。
JVN/NVD/IPAでも利用されています。

#### CCE(Common Configuration Enumeration)

セキュリティ設定項目を一意に識別するID体系です。
OS/ミドルウェアの設定監査で利用されます。

#### CPE(Common Platform Enumeration)

ソフトウェアやOS、ハードウェアを識別するための命名規則です。
「どの製品に該当するか」を機械判定するために必須です。

#### CVSS(Common Vulnerability Scoring System)

脆弱性の深刻度を数値化する指標です。
優先度付けの根拠となり、SCAPでは「評価結果の共通尺度」として使用されます。

#### OVAL(Open Vulnerability and Assessment Language)

システム状態を監査するための記述言語です。
XCCDFが「何を」、OVALが「どう調べるか」という役割です。

#### OCIL(Open Checklist Interactive Language)

自動化できない項目を人に質問するための言語です。
人手確認を標準化しています。

#### ARF(Asset Reporting Format)

評価結果を記録/交換するための形式です。

#### Asset Identification(資産識別)

評価対象システムを識別する仕組みです。

#### Vulnerability Data Feeds(脆弱性情報配信)

NVDなどから提供される脆弱性データです。

#### Checklist Data Feeds(チェックリスト配信)

セキュリティ設定ベンチマークの配信です。

#### SCAP Data Stream

上記要素を一つにまとめて配布する仕組みです。

## MITRE ATT&CK(サイバー攻撃手法を分類したナレッジベース)

### サイバー攻撃手法を分類したナレッジベースとは

**サイバー攻撃手法を分類したナレッジベース** とは、過去に観測・分析された攻撃者の行動や手法を共通の観点・構造で整理し、共有可能にした知識体系のことです。

#### なぜナレッジベースが必要か

従来のセキュリティ対策には次の課題がありました。

- マルウェア名や脆弱性IDごとの **点の対策** にとどまります。
- 攻撃の「手段」だけを見て「目的/流れ」が見えません。
- 組織間での知見共有が困難です。

**攻撃者の行動(TTPs)を軸に整理する必要性** が高まりました。

### MITRE ATT&CK とは

**MITRE ATT&CK** とは、実際のサイバー攻撃事例に基づき、攻撃者の戦術(Tactics)/技術(Techniques)/手順(Procedures)を体系的に整理したナレッジベースです。

#### 公式根拠

- MITRE Corporation
- https://attack.mitre.org/

### MITRE ATT&CK の位置づけ

| 観点 | 内容 |
|---|---|
| 主体 | MITRE(非営利研究機関) |
| 対象 | 実在する攻撃者行動 |
| 目的 | 検知/分析/対策の共通言語化 |
| 利用者 | SOC/CSIRT/ベンダ |

**脆弱性(CVE)ではなく「攻撃行動」を分類する点が本質** です。

### ATT&CK の基本構造(試験最重要)

#### TTPs に基づく整理

ATT&CK は以下の概念に基づきます。

| 要素 | 内容 |
|---|---|
| Tactics | 攻撃の目的(なぜ) |
| Techniques | 攻撃手法(どうやって) |
| Procedures | 実際の実装/ツール |

#### 戦術(Tactics)

戦術は **攻撃ライフサイクル上の目的** を表します。
代表例(Enterprise ATT&CK)は以下のとおりです。

- Initial Access(初期侵入)
- Execution(実行)
- Persistence(永続化)
- Privilege Escalation(権限昇格)
- Defense Evasion(防御回避)
- Credential Access(認証情報窃取)
- Lateral Movement(横展開)
- Command and Control(C2)
- Exfiltration(情報窃取)
- Impact(影響)

**試験では「Initial Access」などの名称を理解しているかが問われます。**

#### 技術(Techniques)

各戦術の下に **具体的な攻撃手法** が定義されています。

例：

- Phishing(T1566)
- Valid Accounts(T1078)
- PowerShell(T1059.001)
- Pass the Hash(T1550.002)

### ATT&CK の3つのマトリクス

MITRE ATT&CK には複数の適用領域があります。

| マトリクス | 対象 |
|---|---|
| Enterprise | 企業IT(Windows/Linux/Cloud) |
| Mobile | モバイル端末 |
| ICS | 制御システム |

### サイバー攻撃ナレッジベースとしての価値

#### 従来型対策との違い

| 従来 | ATT&CK |
|---|---|
| シグネチャ依存 | 行動分析 |
| マルウェア名中心 | 攻撃工程中心 |
| 受動的 | 能動的分析 |

#### 利用例(実務)

- SIEM の検知ルールを ATT&CK 技術にマッピングします。
- EDR アラートの攻撃フェーズを可視化します。
- レッドチーム演習の設計に活用します。
- CSIRT 報告書の共通語彙として使用します。

## ダークネットの概念とセキュリティ上の意義

### ダークネットとは

通常は通信が発生しない、もしくは利用されていないIPアドレス空間のことを指します。

情報セキュリティ分野におけるダークネットとは、実運用されていないIPアドレス群であり、本来正規通信が発生しないアドレス空間です。そこに到達する通信は高い確率で不正活動に由来すると考えられています。

### ダークネット監視の目的

ダークネットを監視する目的は、不正活動の兆候を早期に検知することにあります。
具体的には次のような目的があります。

- マルウェア感染端末の検知
- ワームのスキャン活動の把握
- ボットネットの挙動観測
- 攻撃準備段階の兆候検出

通常通信が存在しないため、少量のパケットでも異常として扱える点が特徴です。

### ダークネットに到達する通信の例

ダークネットに向かう通信には、次のようなものがあります。

- ランダムスキャン型ワームの通信
- 感染端末によるC&Cサーバ探索
- 誤設定されたネットワーク機器の送信
- DDoS攻撃の踏み台探索

これらは、通信の意図そのものが不正または異常である可能性が高いです。

### ダークネットと侵入検知の関係

ダークネットは、IDSやIPSと異なる視点の検知手法です。

- IDS：正規通信の中から異常を検知します。
- ダークネット：本来存在しない通信を検知します。

つまり、ダークネット監視は、誤検知が非常に少ない検知方式といえます。

### ダークネットと類似概念との違い

混同しやすい概念との違いを整理します。

- **ダークネット**：未使用IP空間の監視です。
- **ダークウェブ**：Torなど匿名ネットワーク上の非公開サイトです。
- **ディープウェブ**：認証が必要で検索エンジンに出ないWebです。

### ネットワークテレスコープ

広範囲の未使用IPアドレスを監視し、インターネット全体の不正活動動向を観測する手法です。
研究機関やCERTで活用されます。

### ハニーポットとの違い

- **ハニーポット**：意図的に攻撃を受ける疑似システムです。
- **ダークネット**：何も配置しない空間を監視します。

ダークネットは受動的監視である点が特徴です。

### SOCにおける活用

SOC(Security Operation Center)では、ダークネット監視ログをマルウェア監視の初期兆候として活用します。

## インシデント対応プロセス

### インシデントとは

情報セキュリティにおける**インシデント**とは、情報セキュリティポリシーへの違反、セキュリティ管理策の不具合、またはセキュリティに関連する事象で、事業運営を脅かし情報セキュリティを脅かす可能性のあるものです。

#### インシデントの例

- 不正アクセス（外部からの侵入）
- マルウェア感染
- 情報漏えい（個人情報、営業秘密）
- サービス妨害（DDoS攻撃）
- 内部不正
- Web改ざん

### NIST SP 800-61（インシデント対応ガイドライン）

NIST SP 800-61 Rev.2「Computer Security Incident Handling Guide」は、インシデント対応の国際的な標準ガイドラインです。

#### インシデント対応の4フェーズ

```
1. 準備(Preparation)
   ↓
2. 検知と分析(Detection and Analysis)
   ↓
3. 封じ込め・根絶・復旧(Containment, Eradication, and Recovery)
   ↓
4. 事後活動(Post-Incident Activity)
   ↑_______________(教訓のフィードバック)
```

| フェーズ | 内容 | 主な活動 |
|---------|------|---------|
| **1. 準備** | インシデント発生前の態勢整備 | CSIRTの構築、ツール整備、連絡体制、訓練・演習 |
| **2. 検知と分析** | インシデントの発見と影響分析 | SIEM/IDS/EDRのアラート分析、トリアージ、影響範囲特定 |
| **3. 封じ込め・根絶・復旧** | 被害拡大防止と復旧 | ネットワーク隔離、マルウェア駆除、パッチ適用、システム復旧 |
| **4. 事後活動** | 再発防止と知見の蓄積 | 報告書作成、根本原因分析、ポリシー改善、教訓の共有 |

#### 各フェーズの詳細

##### 1. 準備（Preparation）

- **CSIRT（Computer Security Incident Response Team）**の設置と役割定義
- インシデント対応手順書（プレイブック）の整備
- 連絡体制の構築（経営層、法務、広報、外部機関）
- フォレンジックツール・証拠保全用機材の準備
- 定期的な訓練・演習（机上演習、実動演習）

##### 2. 検知と分析（Detection and Analysis）

- **検知ソース**：SIEM、IDS/IPS、EDR、ファイアウォールログ、ユーザ報告
- **トリアージ**：インシデントの優先度判定（影響範囲×緊急度）
- **分析活動**：IoC（侵害指標）との照合、タイムライン分析、影響範囲の特定
- **誤検知の排除**：False Positiveの識別と除外

##### 3. 封じ込め・根絶・復旧（Containment, Eradication, and Recovery）

| 段階 | 内容 |
|------|------|
| **短期封じ込め** | 感染端末のネットワーク隔離、アカウント無効化 |
| **長期封じ込め** | 暫定的な代替システムの稼働、フォレンジック用証拠保全 |
| **根絶** | マルウェアの完全除去、バックドアの排除、パッチ適用 |
| **復旧** | システムのクリーンインストール、バックアップからの復元、正常性確認 |

##### 4. 事後活動（Post-Incident Activity）

- **事後報告書（Lessons Learned）**の作成
- 根本原因分析（RCA: Root Cause Analysis）
- 対応手順・ポリシーの改善
- 関係機関への報告（個人情報保護委員会、所管省庁等）

### CSIRT（Computer Security Incident Response Team）

#### CSIRTの種類

| 種類 | 説明 |
|------|------|
| **組織内CSIRT** | 自組織のインシデントに対応します |
| **国際連携CSIRT** | 国を代表してインシデント情報を共有します（例：JPCERT/CC） |
| **コーディネーションセンター** | 複数組織間のインシデント対応を調整します |
| **分析センター** | 脅威分析・脆弱性分析を専門に行います |
| **ベンダCSIRT(PSIRT)** | 自社製品の脆弱性・インシデントに対応します |

#### CSIRTとSOCの違い

| 項目 | CSIRT | SOC |
|------|-------|-----|
| 主な役割 | インシデント対応・調整 | セキュリティ監視・検知 |
| 活動タイミング | インシデント発生時（事後対応中心） | 常時監視（24/365） |
| 組織形態 | 兼務が多い | 専任が多い |
| 連携先 | 経営層、法務、外部機関 | CSIRT（検知結果をエスカレーション） |

### インシデント対応における重要概念

#### トリアージ

限られたリソースの中で、**インシデントの優先度を判断**するプロセスです。

| 判断基準 | 内容 |
|---------|------|
| 影響範囲 | 影響を受けるシステム・ユーザの数 |
| 情報の機密性 | 漏えいした情報の機密度 |
| 業務への影響 | 事業継続への支障の程度 |
| 攻撃の進行状況 | 進行中か、完了しているか |

#### エスカレーション

検知した事象を**上位者・関係部門に報告・引き継ぎ**するプロセスです。

- **機能的エスカレーション**：専門チーム（CSIRT等）への引き継ぎ
- **階層的エスカレーション**：経営層・責任者への報告

## サイバーキルチェーン（Cyber Kill Chain）

### サイバーキルチェーンとは

Lockheed Martin社が2011年に提唱した、**標的型攻撃(APT)の攻撃プロセスを7段階で整理したモデル**です。

### 7つの段階

| 段階 | 英語名 | 内容 | 防御のポイント |
|------|--------|------|-------------|
| 1 | **Reconnaissance** | 偵察：標的の情報収集（OSINT等） | 公開情報の管理、SNSポリシー |
| 2 | **Weaponization** | 武器化：エクスプロイト+ペイロードの作成 | （攻撃者側の行為のため防御困難） |
| 3 | **Delivery** | 配送：マルウェアの送付（メール添付/水飲み場等） | メールフィルタ、URLフィルタ、サンドボックス |
| 4 | **Exploitation** | 攻撃：脆弱性を突いてコード実行 | パッチ管理、脆弱性管理 |
| 5 | **Installation** | インストール：バックドア/RAT等の永続化 | EDR、ホワイトリスト制御 |
| 6 | **Command and Control** | C2通信：攻撃者との通信確立 | プロキシ監視、DNS監視、出口対策 |
| 7 | **Actions on Objectives** | 目的の実行：情報窃取・破壊等 | DLP、暗号化、アクセス制御 |

### キルチェーンの活用方法

- **早い段階での検知・遮断が重要**（左側で止めるほど被害が小さい）
- 各段階に対応する**防御策のマッピング**に活用します
- ATT&CKはキルチェーンの各段階をさらに詳細化したものです

### キルチェーンとATT&CKの関係

| サイバーキルチェーン | ATT&CK Tactics（対応例） |
|--------------------|------------------------|
| Reconnaissance | Reconnaissance |
| Weaponization | Resource Development |
| Delivery | Initial Access |
| Exploitation | Execution |
| Installation | Persistence, Defense Evasion |
| C2 | Command and Control |
| Actions on Objectives | Exfiltration, Impact |

## 脅威情報共有（Threat Intelligence）

### 脅威インテリジェンスとは

サイバー攻撃に関する**情報を収集・分析・共有し、組織の防御に活用する**取り組みです。

### 脅威インテリジェンスの4つのレベル

| レベル | 内容 | 利用者 |
|--------|------|--------|
| **戦略的(Strategic)** | 脅威の動向・リスク評価 | 経営層 |
| **戦術的(Tactical)** | 攻撃者のTTPs | セキュリティアナリスト |
| **運用的(Operational)** | 特定攻撃キャンペーンの詳細 | CSIRT/SOC |
| **技術的(Technical)** | IoC（IPアドレス、ハッシュ値等） | セキュリティ機器 |

### IoC（Indicators of Compromise：侵害指標）

攻撃が行われた、または行われている**証拠となる技術的な指標**です。

| IoC の種類 | 例 |
|-----------|------|
| IPアドレス | C2サーバのIPアドレス |
| ドメイン名 | フィッシングサイトのドメイン |
| ファイルハッシュ | マルウェアのSHA-256ハッシュ値 |
| URL | マルウェア配布URL |
| メール送信元 | フィッシングメールのFrom |
| レジストリキー | マルウェアが作成する永続化キー |

- IoC は**SIEM/EDR/ファイアウォール**等に投入して自動検知に活用します
- 鮮度が重要であり、攻撃者はIoCを頻繁に変更します

### STIX（Structured Threat Information eXpression）

**脅威情報を構造化して記述するための標準フォーマット**です（OASIS標準）。

#### STIX 2.1 のオブジェクト

| オブジェクト | 内容 |
|------------|------|
| **Attack Pattern** | 攻撃パターン（ATT&CK Techniqueとの連携） |
| **Campaign** | 攻撃キャンペーン |
| **Indicator** | IoC（侵害指標） |
| **Malware** | マルウェア情報 |
| **Threat Actor** | 脅威アクター（攻撃者グループ） |
| **Vulnerability** | 脆弱性情報 |
| **Relationship** | オブジェクト間の関係性 |

- JSON形式で記述されます
- 組織間での脅威情報の相互運用性を確保します

### TAXII（Trusted Automated eXchange of Intelligence Information）

**STIX形式の脅威情報を自動的に交換するための通信プロトコル**です。

| 項目 | 内容 |
|------|------|
| 通信方式 | HTTPS(REST API) |
| データ形式 | STIX 2.1 |
| 交換モデル | **Collection**（プル型）、**Channel**（プッシュ型） |
| 用途 | 組織間・ISACでの脅威情報自動共有 |

### STIX/TAXII の関係

```
組織A → STIX形式で脅威情報を作成
         ↓ TAXIIプロトコルで送信
共有プラットフォーム(ISAC等)
         ↓ TAXIIプロトコルで配信
組織B → STIX形式の脅威情報を受信 → SIEM/EDRに投入
```

### その他の脅威情報共有の枠組み

| 名称 | 内容 |
|------|------|
| **ISAC** | 業界別の情報共有組織（金融ISAC、ICT-ISAC等） |
| **TLP（Traffic Light Protocol）** | 情報共有範囲を色で示す分類（RED/AMBER/GREEN/WHITE） |
| **MISP** | オープンソースの脅威情報共有プラットフォーム |

## 脆弱性管理ライフサイクル

### 脆弱性管理とは

組織が保有するIT資産の脆弱性を**継続的に発見・評価・対応・監視**するプロセスです。

### 脆弱性管理の4つのフェーズ

```
1. 発見(Discovery)
   ↓
2. 評価(Assessment)
   ↓
3. 対応(Remediation)
   ↓
4. 監視(Monitoring)
   ↑_________(継続的サイクル)
```

| フェーズ | 内容 | 主な手法 |
|---------|------|---------|
| **1. 発見** | 脆弱性の検出 | 脆弱性スキャナ、脅威インテリジェンス、JVN/NVD監視 |
| **2. 評価** | 深刻度・優先度の判断 | CVSS、SSVC、資産重要度の考慮 |
| **3. 対応** | 脆弱性の修正・緩和 | パッチ適用、設定変更、WAFルール追加、リスク受容 |
| **4. 監視** | 対応状況の追跡と再評価 | ダッシュボード、KPI管理、再スキャン |

### SSVC（Stakeholder-Specific Vulnerability Categorization）

CISA(米国サイバーセキュリティ・インフラストラクチャセキュリティ庁)が推奨する、**CVSSを補完する脆弱性対応優先度の決定フレームワーク**です。

#### SSVCの決定木（Decision Tree）

SSVCは以下の観点でトリアージを行い、**4段階のアクション**を決定します。

| 判断基準 | 内容 |
|---------|------|
| **Exploitation** | 実際に悪用されているか（Active/PoC/None） |
| **Automatable** | 攻撃の自動化が可能か |
| **Technical Impact** | 技術的影響の大きさ（Partial/Total） |
| **Mission Prevalence** | 組織のミッションへの影響度 |

#### SSVCのアクション（出力）

| アクション | 意味 |
|-----------|------|
| **Act** | 即座に対応が必要です |
| **Attend** | 早期に対応を検討します |
| **Track*** | 通常の管理サイクルで対応します |
| **Track** | 通常の管理で監視継続します |

#### CVSSとSSVCの違い

| 項目 | CVSS | SSVC |
|------|------|------|
| 評価対象 | 脆弱性の深刻度（技術的） | 対応の優先度（組織的） |
| 出力 | スコア（0.0～10.0） | アクション（Act/Attend/Track等） |
| 考慮要素 | 攻撃容易性、影響度 | 実際の悪用状況、組織への影響 |
| 目的 | 深刻度の共通尺度 | 対応判断の意思決定支援 |

### パッチ管理（Patch Management）

#### パッチ適用のプロセス

```
脆弱性情報の収集 → 影響評価 → テスト環境で検証 → 適用計画策定 → 本番適用 → 確認
```

#### パッチ適用が困難な場合の緩和策

| 緩和策 | 内容 |
|--------|------|
| **仮想パッチ** | WAF/IPSのルールで脆弱性を狙う通信を遮断します |
| **ネットワーク分離** | 脆弱なシステムをネットワーク的に隔離します |
| **アクセス制御強化** | 脆弱なサービスへのアクセスを制限します |
| **リスク受容** | 残留リスクを文書化し、経営層が受容を判断します |

## デジタルフォレンジックスと証拠保全

### デジタルフォレンジックスとは

**デジタルフォレンジックス(Digital Forensics)** とは、コンピュータやネットワーク等の電子的記録を **法的証拠として利用可能な形で収集/保全/解析/報告する技術および手続** です。

#### 主な定義元

- NIST SP 800-86
- ACPO Guidelines(英警察)
- IPA「デジタル/フォレンジック/ガイドライン」

### デジタルフォレンジックスの目的

| 観点 | 内容 |
|---|---|
| 技術的目的 | インシデント原因/侵入経路の特定 |
| 法的目的 | 証拠能力を保持した証拠提出 |
| 組織的目的 | 再発防止/内部統制 |

**「調査」ではなく「証拠化」が本質** である点が重要です。

### フォレンジックスで最重要となる考え方

#### 証拠能力の維持

証拠は以下を満たす必要があります。

- 完全性(改ざんされていない)
- 再現性(第三者が検証可能)
- 追跡性(誰が/いつ/何をしたか)

これを支えるのが **証拠保全の原則** です。

### 証拠の滅失を避けるための原則(最重要)

#### 原則の背景

デジタルデータは、電源断・再起動・通信切断などにより **瞬時に消失/変化** します。
そのため、**誤った手順＝証拠能力の喪失** につながります。

#### 基本原則(ACPO/IPA)

##### 原則(1)：原本に手を加えない

- 原本データは変更しません。
- 作業は必ず **複製(イメージ)** に対して実施します。

##### 原則(2)：完全性を証明できること

- ハッシュ値(SHA-256等)で改ざん検証します。
- 取得前後でハッシュ一致を確認します。

##### 原則(3)：第三者が検証可能であること

- 手順書/ログを残します。
- 属人化しません。

##### 原則(4)：責任の所在を明確にする

- 誰が証拠を扱ったか(Chain of Custody)を記録します。

### 証拠保全の順番(Order of Volatility)

#### なぜ「順番」が重要か

デジタル証拠には **揮発性の違い** があります。
消えやすいものから優先的に保全する必要があります。

#### 証拠保全の代表的な順序(試験頻出)

以下は **IPA/NIST で広く採用されている順序** です。

| 優先度 | 証拠例 |
|---|---|
| (1) 最優先 | CPUレジスタ、キャッシュ |
| (2) | メインメモリ(RAM) |
| (3) | ネットワーク接続情報、通信セッション |
| (4) | 実行中プロセス、ログインユーザ |
| (5) | ディスク上のログ、ファイル |
| (6) | バックアップ媒体 |

**ポイント：「電源を切る前にメモリを取る」**
**参考URL：[RFC3227 証拠収集とアーカイビングのためのガイドライン](https://www.nic.ad.jp/ja/tech/ipa/RFC3227JA.html)**

#### 試験で狙われる誤り例

- まず電源を切る → RAM 情報が消失します。
- OS標準コマンドで操作 → 証拠改変の疑いが生じます。

### Chain of Custody(証拠管理の連鎖)

**Chain of Custody** とは、証拠が取得されてから提出されるまでの管理/移動/取扱の記録のことです。

#### 記録すべき情報

- 証拠ID
- 取得日時
- 取得者
- 保管場所
- 取扱履歴

### デジタルフォレンジックスの一般的プロセス

| フェーズ | 内容 |
|---|---|
| 収集 | 証拠取得(揮発性優先) |
| 保全 | ハッシュ計算/封印 |
| 解析 | 不正操作/侵入経路分析 |
| 報告 | 時系列/因果関係整理 |

### ライブフォレンジックスとデッドフォレンジックス

#### ライブフォレンジックス

- 稼働中システムから証拠取得します。
- メモリ/通信情報が対象です。
- 証拠改変リスクがあるため、熟練が必要です。

#### デッドフォレンジックス

- 電源断後の記憶媒体解析です。
- 再現性が高いです。
- 揮発情報は取得不可です。

### ハッシュアルゴリズム

| アルゴリズム | 用途 |
|---|---|
| SHA-256 | 証拠完全性確認 |
| SHA-1 | 非推奨 |
| MD5 | 衝突のため不可 |

## 脆弱性検査とポートスキャン

### 脆弱性検査(Vulnerability Assessment)

システムの弱点(脆弱性)を特定するために、ネットワークスキャン/サービス列挙/設定診断/既知脆弱性の検査などを体系的に行うプロセスです。

#### 脆弱性検査の主要手法

- ネットワークスキャン(Port Scanning/Service Scanning)
- ホスト設定評価(Configuration Review)
- アプリケーション脆弱性検査(Web/API)
- パッチ適用状況の確認
- 脆弱性スキャナ(例：Nessus/OpenVAS)の使用

### ポートスキャン(Port Scanning)

ターゲットホストのポート(TCP/UDP)に対し、接続要求またはパケットを送り、開いているポート(Open)/閉じているポート(Closed)/フィルタリングされているポートを識別する行為です。
**ポートスキャンは脆弱性検査の基礎であり、攻撃者の偵察の第一段階** です。

#### 代表的なポートスキャン手法

1. **TCP Connect Scan(フルコネクトスキャン)**
    - OSが行う通常の3ウェイハンドシェイクを実行します。
    - 最も確実ですが目立ちます。
2. **SYN Scan(半開スキャン/Stealth Scan)**
    - SYN → SYN/ACK(Open)で判断します。
    - 完全接続しないためステルス性が高いです。
    - Nmapのデフォルト手法です。
3. **UDP Scan**
    - UDPは応答が返却されないため難しいです。
    - ICMP Port UnreachableでCloseを判定します。
4. **FIN/Xmas/Nullスキャン**
    - FW回避のために使われます。
    - 現代のOSでは対策されていることが多いです。

#### ポートスキャンでわかる情報

- 稼働中のサービス(例：22/ssh, 80/http, 443/https)
- 不要なサービスの発見
- ファイアウォールの設定状況
- 外部公開されてはいけない管理ポート(Telnet/RDP等)
- 攻撃対象となりうるソフトウェア(Apache/MySQLなど)

#### ポートスキャンの結果から判明する攻撃対象

開いているポートがわかることで、攻撃対象の入り口となります。

- 80/tcp：XSS, SQLi, OSコマンドインジェクション
- 22/tcp：SSHブルートフォース
- 445/tcp：SMB脆弱性(EternalBlue)
- 3389/tcp：RDP強制攻撃
- 161/udp：SNMP情報漏えい

### 脆弱性スキャナ(Vulnerability Scanner)の仕組み

#### 主なスキャナ

- Nessus
- OpenVAS
- QualysGuard
- Nikto(Web)
- Burp Suite(Webアプリ)

#### 実施内容

- ポートスキャン
- サービスバナー取得(Banner Grabbing)
- バージョン判定
- 既知脆弱性DB(CVE/NVD)との照合
- 設定ミス(Misconfiguration)検査
- 弱いパスワードの試行

## 周辺知識

### CVE(Common Vulnerabilities and Exposures)

- MITRE が管理する **脆弱性の識別子** です。
- 例：CVE-2021-44228(Log4Shell)

### CVE/CWE/ATT&CK の違い

| 分類 | 何を扱うか |
|---|---|
| CVE | 脆弱性の識別子 |
| CWE | 脆弱性の設計上の弱点 |
| ATT&CK | 攻撃者の行動 |

### Kill Chain との関係

- Cyber Kill Chain：攻撃の段階モデルです。
- ATT&CK：段階ごとの具体的行動です。

**ATT&CK は Kill Chain を詳細化したものと理解できます。**

### DEFEND/D3FEND

- MITRE D3FEND：防御側の対抗手段を整理したものです。
- ATT&CK と対になる概念です。

### CERT/CC

- 米国のコンピュータセキュリティインシデント対応機関です。
- 脆弱性調整プロセスの先駆者です。

### JVN/NVD との関係

- JVN：日本の脆弱性情報です。
- NVD：米国NISTの脆弱性情報です。
- 両者とも **CVSS Base Score を掲載** しています。

### OSフィンガープリント(OS Fingerprinting)

「SYN/ACKのTTL値」や「応答フラグの違い」からOSを推測する技術です。

### Banner Grabbing(バナー取得)

```TEXT
$ nc example.com 80
HTTP/1.1 200 OK
Server: Apache/2.4.1
```

サーバのバージョン等が漏れることにより、該当バージョンの脆弱性が特定されてしまいます。

### ファイアウォール越えのスキャン

- Idle Scan(ゾンビホストを利用)
- Fragmentation(パケット断片化)
- Source Port Manipulation(DNSポート53を偽装)

### ポートステート分類(Nmap)

| 状態 | 意味 |
|:-----|:-----|
| Open | 受信し応答あり(サービス稼働中) |
| Closed | 応答はあるがサービスなし |
| Filtered | FWにより遮断、応答なし |
| Unfiltered | 応答はあるが状態不明 |

### 脆弱性検査とペネトレーションテストとの違い

下記表のとおり、脆弱性検査は「安全に弱点を洗い出す」ものであり、ペネトレーションテストは「実際に侵入できるかを確認する」ものです。

| 種類 | 目的 | 内容 |
|:-----|:-----|:-----|
| 脆弱性検査(VA) | 弱点を見つける | スキャン中心、非破壊 |
| 侵入テスト(Penetration Test) | 実際に突破できるか | 攻撃を実行(制御された攻撃) |

### ペネトレーションテストの分類

| 分類 | 知識範囲 | 特徴 |
|------|---------|------|
| **ブラックボックステスト** | 内部情報なし | 外部攻撃者の視点を再現します |
| **ホワイトボックステスト** | 全情報あり | 効率的だが現実性は低いです |
| **グレーボックステスト** | 一部情報あり | 内部犯行者/委託先の視点です |

### TLPT（Threat-Led Penetration Testing：脅威ベースペネトレーションテスト）

実在する脅威シナリオに基づき、**組織の検知・対応能力を実践的に評価する**テスト手法です。

#### TLPTの特徴

| 項目 | 従来のペネトレーションテスト | TLPT |
|------|------------------------|------|
| シナリオ | 汎用的な攻撃手法 | **実在する脅威アクターのTTPsに基づく** |
| 対象 | 技術的な脆弱性 | 技術＋人＋プロセス（検知・対応含む） |
| 評価対象 | 侵入可否 | **検知能力・対応能力を含む総合評価** |
| 関連フレームワーク | OSSTMM、PTES | **TIBER-EU**、CBEST、TLPT実践ガイド |

- 金融庁が「金融分野におけるTLPT実践ガイド」を公表しています
- **レッドチーム**（攻撃側）と**ブルーチーム**（防御側）に分かれて実施します
- ブルーチームにはテスト実施を**事前に知らせない**のが原則です

### レッドチーム・ブルーチーム・パープルチーム

| チーム | 役割 |
|--------|------|
| **レッドチーム** | 攻撃者を模擬し、組織の防御を突破する活動を行います |
| **ブルーチーム** | 攻撃の検知・対応を行う防御側チームです |
| **パープルチーム** | レッド・ブルーの両方の視点で協調し、防御力向上を図ります |

### バグバウンティ（Bug Bounty）

組織が**外部の研究者に脆弱性の発見・報告を奨励し、報奨金を支払う**制度です。

- 従来のペネトレーションテストを補完する仕組みです
- 多様な視点・スキルによる継続的な脆弱性発見が可能です
- 責任ある開示（Coordinated Disclosure）が前提です

## まとめ

### SCAP構成要素の一覧

| 要素 | 正式名称 | 役割 |
|------|---------|------|
| CVE | Common Vulnerabilities and Exposures | 脆弱性の一意な識別ID |
| CWE | Common Weakness Enumeration | 脆弱性の根本原因（弱点）の分類 |
| CCE | Common Configuration Enumeration | セキュリティ設定項目の識別ID |
| CPE | Common Platform Enumeration | ソフトウェア/OS/HWの命名規則 |
| CVSS | Common Vulnerability Scoring System | 脆弱性深刻度のスコアリング |
| OVAL | Open Vulnerability and Assessment Language | システム状態の監査記述言語 |
| XCCDF | eXtensible Configuration Checklist Description Format | セキュリティ設定チェックリスト |

### CVSS バージョン比較

| 項目 | CVSS v3.1 | CVSS v4.0 |
|------|-----------|-----------|
| メトリクスグループ | Base/Temporal/Environmental | Base/Threat/Environmental/**Supplemental** |
| Scope | あり | **廃止**（脆弱システム/後続システムに分離） |
| User Interaction | 2段階(None/Required) | **3段階**(None/Passive/Active) |
| Attack Requirements | なし | **新設(AT)** |
| Temporal指標 | E/RL/RCの3指標 | **Exploit Maturity(E)のみ** |
| スコア種類 | 1種類 | **4種類**(CVSS-B/BT/BE/BTE) |

### 脆弱性情報源の比較

| 情報源 | 運営 | 範囲 | 特徴 |
|--------|------|------|------|
| **JVN** | IPA + JPCERT/CC | 国内中心 | 調整済み脆弱性情報の公表 |
| **JVN iPedia** | IPA | 国内外 | 脆弱性データベース（検索可能） |
| **NVD** | NIST（米国） | 全世界 | CVE/CVSS/CPEを含む包括的DB |

| **CWE** | MITRE | 分類体系 | 脆弱性の根本原因の分類 |

| 優先度 | 対象 | 揮発性 | 保全方法 |
|--------|------|--------|---------|
| 1（最優先） | CPUレジスタ/キャッシュ | 最高 | メモリダンプ |
